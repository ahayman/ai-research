Why On-Device Emotion Detection?

Shipping emotion detection directly on the user's iPhone or Mac eliminates network latency, removes cloud API costs, and keeps sensitive audio data entirely on the device. For a transcription app, this means you can annotate each utterance with an emotion score in real time, without ever sending a single audio frame to a server.

Key Advantage
On-device processing means raw audio never leaves the device. This dramatically simplifies GDPR, HIPAA, and EU AI Act compliance for emotion-related features.

Apple's stack provides every piece you need: AVAudioEngine captures microphone audio, Core ML runs your emotion model on the Neural Engine, Sound Analysis offers a turnkey classifier pipeline, and the Foundation Models framework gives you an on-device LLM for text-based emotion analysis of your transcriptions.

This page is organized as a practical build guide. We start with the full system architecture, walk through each Apple framework, cover model conversion and compression, and finish with complete Swift code you can adapt for your own app.

Critical Threading Rule
Never run Core ML inference on the audio render thread. The audio tap callback must return within microseconds. Copy the buffer and dispatch to a background queue immediately.

Core ML

Core ML is the inference engine at the center of Apple's ML stack. It loads a compiled model, optimally dispatches operations to the Neural Engine, GPU, or CPU, and returns predictions through a Swift-native API.

Model Formats

FormatExtensionSpecStatus
Neural Network.mlmodelProtobuf-based, fixed layer graphLegacy (maintained)
ML Program.mlpackageTyped, programmatic executionRecommended for all new models

Always target .mlpackage (ML Program) for new work. It gets all new features, supports modern operators needed by transformer-based emotion models, and plays well with the Neural Engine.

Inference API

Core ML provides both synchronous and async prediction APIs. The async variant (introduced at WWDC 2023) is thread-safe and integrates with Swift concurrency.

SwiftCore ML Async Prediction
import CoreML

class EmotionPredictor {
    private let model: MLModel

    init() throws {
        let config = MLModelConfiguration()
        config.computeUnits = .all  // Let Core ML pick best hardware
        self.model = try EmotionClassifier(configuration: config).model
    }

    func predict(features: MLMultiArray) async throws -> [String: Double] {
        let input = try MLDictionaryFeatureProvider(
            dictionary: ["audio_features": MLFeatureValue(multiArray: features)]
        )

        // Async prediction — thread-safe, supports cancellation
        let result = try await model.prediction(from: input)

        // Extract emotion probabilities
        guard let probs = result.featureValue(
            for: "emotionProbabilities"
        )?.dictionaryValue as? [String: NSNumber] else {
            throw EmotionError.invalidOutput
        }

        return probs.mapValues { $0.doubleValue }
    }
}

Input Types for Audio

Audio emotion models typically accept one of these input formats:

  • MLMultiArray — raw tensor data (mel spectrogram, MFCC features). Shape might be [1, 128, 64] for a 128-bin mel spectrogram with 64 time frames.
  • MLShapedArray — ergonomic Swift wrapper over MLMultiArray with typed subscripts. Fully interconvertible.
  • SoundAnalysisPreprocessing — built-in Core ML layer that takes raw audio samples and outputs mel spectrograms directly inside the model (no external feature extraction needed).

Sound Analysis Framework

Apple's Sound Analysis framework provides a high-level pipeline for audio classification. It handles windowing, buffering, and result delivery — you just provide a model and audio stream.

Built-in vs. Custom Classifiers

Built-in Classifier

  • 300+ sound categories pre-trained
  • Includes: speech, laughter, crying, shouting, sighing
  • Window: 0.5s – 15s (recommended ≥ 1s)
  • Zero model size — built into OS
  • Good starting point for coarse emotion cues

Custom Classifier

  • Your Core ML model for fine-grained emotions
  • Trained on emotion-labeled speech data
  • Plug-and-play with Sound Analysis pipeline
  • Full control over categories and accuracy
  • Can run alongside built-in classifier

Real-Time Stream Analysis

SwiftSound Analysis with Custom Model
import SoundAnalysis
import AVFoundation

class EmotionSoundAnalyzer: NSObject, SNResultsObserving {
    private let engine = AVAudioEngine()
    private var analyzer: SNAudioStreamAnalyzer?

    func startAnalysis() throws {
        let inputNode = engine.inputNode
        let recordingFormat = inputNode.outputFormat(forBus: 0)

        // Create stream analyzer matching the audio format
        analyzer = SNAudioStreamAnalyzer(format: recordingFormat)

        // Load your custom emotion Core ML model
        let emotionModel = try EmotionClassifier(configuration: MLModelConfiguration()).model
        let request = try SNClassifySoundRequest(mlModel: emotionModel)
        request.windowDuration = CMTimeMakeWithSeconds(1.5, preferredTimescale: 48_000)
        request.overlapFactor = 0.5

        try analyzer?.add(request, withObserver: self)

        // Install tap and feed buffers to analyzer
        inputNode.installTap(onBus: 0, bufferSize: 8192, format: recordingFormat) {
            [weak self] buffer, time in
            self?.analyzer?.analyze(buffer, atAudioFramePosition: time.sampleTime)
        }

        try engine.start()
    }

    // Called when classification results are ready
    func request(_ request: SNRequest, didProduce result: SNResult) {
        guard let classification = result as? SNClassificationResult else { return }

        let emotions = classification.classifications.map {
            (label: $0.identifier, confidence: $0.confidence)
        }

        // Dispatch to main thread for UI update
        DispatchQueue.main.async {
            self.updateUI(with: emotions)
        }
    }

    func request(_ request: SNRequest, didFailWithError error: Error) {
        print("Analysis error: \(error)")
    }
}

Create ML: Training Your Own Emotion Classifier

Create ML's MLSoundClassifier is the fastest way to train a custom emotion model that runs natively on Apple devices.

Training Steps

1

Organize Audio Data

Create folders by emotion label: angry/, happy/, sad/, neutral/, fearful/. Place 50+ clips per category, 3–10 seconds each, at 16 kHz.

2

Open Create ML & Select Sound Classification

In Xcode, go to Xcode → Open Developer Tool → Create ML. Choose the Sound Classification template.

3

Train

Drag your training folder in and click Train. Create ML handles all audio preprocessing (spectral feature extraction) and uses transfer learning from Apple's audio backbone.

4

Evaluate & Export

Review accuracy on the validation set. If acceptable, drag the model out of Create ML and drop it into your Xcode project. It becomes a .mlmodel that Xcode auto-compiles.

Tip: Data Augmentation
Augment your dataset with pitch shifting (±2 semitones), time stretching (0.9x–1.1x), and background noise addition. This dramatically improves generalization for emotion recognition.

AVAudioEngine: Real-Time Audio Capture

AVAudioEngine is the gateway to the microphone. It provides a node graph where you can capture, resample, and tap audio buffers for ML processing.

Format Conversion

The microphone input node always outputs at the hardware's native format (typically 44.1 or 48 kHz, stereo, float32). Most emotion models expect 16 kHz mono. You must resample:

SwiftAVAudioEngine Setup with Resampling
import AVFoundation

class AudioCapture {
    private let engine = AVAudioEngine()
    private let processingQueue = DispatchQueue(
        label: "com.app.audioProcessing",
        qos: .userInitiated
    )

    // Target format for emotion model
    private let targetFormat = AVAudioFormat(
        commonFormat: .pcmFormatFloat32,
        sampleRate: 16000,
        channels: 1,
        interleaved: false
    )!

    func startCapture(onBuffer: @escaping (AVAudioPCMBuffer) -> Void) throws {
        let inputNode = engine.inputNode
        let hwFormat = inputNode.outputFormat(forBus: 0)

        // Insert a mixer node for format conversion
        let mixer = AVAudioMixerNode()
        engine.attach(mixer)
        engine.connect(inputNode, to: mixer, format: hwFormat)

        // Tap the mixer at our target format (16 kHz mono)
        mixer.installTap(
            onBus: 0,
            bufferSize: 4096,  // ~256 ms at 16 kHz
            format: targetFormat
        ) { [weak self] buffer, time in
            // Immediately dispatch off the audio thread
            self?.processingQueue.async {
                onBuffer(buffer)
            }
        }

        engine.prepare()
        try engine.start()
    }

    func stopCapture() {
        engine.inputNode.removeTap(onBus: 0)
        engine.stop()
    }
}

Apple Neural Engine (ANE)

The Apple Neural Engine is a dedicated ML accelerator in every modern Apple chip. When Core ML routes your emotion model to the ANE, you get fast inference at minimal battery cost — ideal for continuous real-time analysis.

ANE-Friendly Operations

Runs Well on ANE

  • Convolutions (1D, 2D, depthwise)
  • Matrix multiplications
  • ReLU, GELU, sigmoid, tanh
  • Batch norm, layer norm
  • Pooling (avg, max)
  • Softmax, element-wise ops
  • Transformer attention (ML Programs)

Falls Back to CPU/GPU

  • Custom layers (no public ANE API)
  • Some broadcastable layer types
  • Dynamic tensor shapes
  • Very large models (>500 MB)
  • Exotic/unusual operators
  • Complex control flow

ANE Optimization Checklist

  1. Use static shapes — fix your input dimensions (e.g., [1, 128, 64] for mel spectrogram)
  2. Target ML Program format — convert with convert_to="mlprogram"
  3. Apply palettized weights (4–8 bit) — reduces memory bandwidth on ANE
  4. Replace unsupported layers — model surgery to swap broadcastable ops for ANE-friendly equivalents
  5. Profile with Instruments — Core ML Instrument shows exactly which compute unit each operation uses
  6. Split if needed — first part on ANE, last few layers on CPU if they contain unsupported ops

Conversion Code

Pythoncoremltools Conversion
import coremltools as ct
import torch

# 1. Load your trained PyTorch emotion model
model = EmotionCNN()
model.load_state_dict(torch.load("emotion_model.pt"))
model.eval()

# 2. Trace with representative input
#    Shape: [batch, mel_bins, time_frames]
example_input = torch.randn(1, 128, 64)
traced = torch.jit.trace(model, example_input)

# 3. Convert to Core ML ML Program
mlmodel = ct.convert(
    traced,
    inputs=[ct.TensorType(
        name="audio_features",
        shape=(1, 128, 64),
        dtype=ct.converters.mil.mil.types.fp32
    )],
    outputs=[ct.TensorType(name="emotionProbabilities")],
    convert_to="mlprogram",
    minimum_deployment_target=ct.target.iOS17,
)

# 4. Add metadata
mlmodel.author = "Your App"
mlmodel.short_description = "Speech emotion classifier (7 emotions)"
mlmodel.input_description["audio_features"] = "128-bin mel spectrogram, 64 time frames (~2 sec)"

# 5. Save
mlmodel.save("EmotionClassifier.mlpackage")
Conversion Gotcha
Wav2vec2 and HuBERT models use dynamic shapes and complex attention patterns. You may need to fix sequence lengths, replace unsupported ops, or use torch.onnx.export as an intermediate step. Test thoroughly — verify Core ML output matches PyTorch output within acceptable tolerance.

WhisperKit

WhisperKit brings OpenAI Whisper to Apple devices with native ANE acceleration. For a transcription app, it is the primary on-device ASR solution.

Architecture

  • Pure Swift, distributed as a Swift Package
  • Audio Encoder reimplemented for streaming inference on ANE
  • Text Decoder works with partial audio for real-time output
  • Whisper Large v3 Turbo (1B params) fits on-device
  • Pre-converted Core ML models on HuggingFace (argmaxinc/whisperkit-coreml)

Adding Emotion Detection

WhisperKit's encoder produces rich audio representations that already capture prosodic and tonal information. You can add an emotion classification head:

  1. Extract encoder features — take the output of Whisper's audio encoder (before the decoder)
  2. Add a classification head — a small MLP or attention pooling layer + linear classifier
  3. Fine-tune on emotion data — freeze the encoder, train only the head on labeled emotion data
  4. Export both models — shared encoder feeds both decoder (transcription) and emotion head
Trade-off
Sharing the Whisper encoder between transcription and emotion detection saves memory and compute, but couples the two tasks. A separate lightweight emotion model (5–15 MB) running independently may be simpler to maintain and update.

Apple Foundation Models Framework

Introduced at WWDC 2025, the Foundation Models framework gives you an on-device LLM for free — no API key, no internet required. It runs on the Neural Engine and is optimized for Apple Silicon.

Emotion Analysis from Transcribed Text

While your Core ML model detects emotion from audio acoustics (prosody, pitch, energy), the Foundation Models framework can analyze the semantic content of your transcription for a complementary text-based emotion signal.

SwiftFoundation Models Emotion Analysis
import FoundationModels

// Define structured output
struct EmotionAnalysis: Generable {
    var primaryEmotion: String
    var confidence: Double
    var valence: Double     // -1 (negative) to 1 (positive)
    var arousal: Double     // 0 (calm) to 1 (excited)
}

func analyzeTextEmotion(transcript: String) async throws -> EmotionAnalysis {
    let session = LanguageModelSession(
        instructions: """
        Analyze the emotional content of the given speech transcript.
        Classify the primary emotion and estimate valence/arousal scores.
        """
    )

    let result = try await session.respond(
        to: transcript,
        generating: EmotionAnalysis.self
    )
    return result
}

Multimodal Fusion

Combine the acoustic emotion score (from Core ML) with the semantic emotion score (from Foundation Models) for a more robust result:

SwiftCombining Audio + Text Emotion
struct FusedEmotion {
    let acoustic: [String: Double]  // from Core ML audio model
    let semantic: EmotionAnalysis     // from Foundation Models

    var finalEmotion: String {
        // Weight acoustic features higher (0.6) as they capture
        // tone/prosody that text alone misses
        let acousticWeight = 0.6
        let semanticWeight = 0.4

        // Combine scores (simplified — real impl uses full probability distributions)
        var combined = acoustic.mapValues { $0 * acousticWeight }
        combined[semantic.primaryEmotion, default: 0] += semantic.confidence * semanticWeight

        return combined.max(by: { $0.value < $1.value })!.key
    }
}

Complete Swift Implementation

Here is a production-ready architecture that ties everything together: audio capture, feature extraction, Core ML inference, and optional Foundation Models text analysis.

EmotionDetectionManager

SwiftEmotionDetectionManager.swift
import AVFoundation
import CoreML
import Accelerate

/// Manages the complete audio emotion detection pipeline.
/// Audio capture → Feature extraction → Core ML inference → Results
@MainActor
final class EmotionDetectionManager: ObservableObject {

    // MARK: - Published State
    @Published var currentEmotions: [String: Double] = [:]
    @Published var dominantEmotion: String = "neutral"
    @Published var isAnalyzing: Bool = false

    // MARK: - Audio Engine
    private let engine = AVAudioEngine()
    private let mixer = AVAudioMixerNode()

    // MARK: - Processing
    private let processingQueue = DispatchQueue(
        label: "emotion.processing", qos: .userInitiated
    )
    private let mlQueue = DispatchQueue(
        label: "emotion.ml", qos: .userInitiated
    )

    // MARK: - Model
    private var emotionModel: MLModel?
    private let sampleRate: Double = 16000
    private let windowDuration: Double = 2.0   // seconds
    private let hopDuration: Double = 1.0      // 50% overlap
    private var audioBuffer: [Float] = []
    private let windowSize: Int  // sampleRate * windowDuration

    // MARK: - Mel Spectrogram Parameters
    private let fftSize = 512
    private let hopLength = 256
    private let melBins = 128

    init() {
        windowSize = Int(sampleRate * windowDuration)
        loadModel()
    }

    private func loadModel() {
        do {
            let config = MLModelConfiguration()
            config.computeUnits = .all
            emotionModel = try EmotionClassifier(configuration: config).model
        } catch {
            print("Failed to load emotion model: \(error)")
        }
    }

    // MARK: - Start / Stop

    func start() throws {
        let inputNode = engine.inputNode
        let hwFormat = inputNode.outputFormat(forBus: 0)

        engine.attach(mixer)
        engine.connect(inputNode, to: mixer, format: hwFormat)

        let targetFormat = AVAudioFormat(
            commonFormat: .pcmFormatFloat32,
            sampleRate: sampleRate,
            channels: 1,
            interleaved: false
        )!

        mixer.installTap(
            onBus: 0, bufferSize: 4096, format: targetFormat
        ) { [weak self] buffer, _ in
            self?.processingQueue.async {
                self?.handleAudioBuffer(buffer)
            }
        }

        engine.prepare()
        try engine.start()
        isAnalyzing = true
    }

    func stop() {
        mixer.removeTap(onBus: 0)
        engine.stop()
        isAnalyzing = false
        audioBuffer.removeAll()
    }

    // MARK: - Buffer Processing

    private func handleAudioBuffer(_ buffer: AVAudioPCMBuffer) {
        guard let channelData = buffer.floatChannelData?[0] else { return }
        let frameCount = Int(buffer.frameLength)

        // Accumulate samples
        audioBuffer.append(contentsOf:
            UnsafeBufferPointer(start: channelData, count: frameCount)
        )

        // Process when we have enough for a window
        while audioBuffer.count >= windowSize {
            let window = Array(audioBuffer.prefix(windowSize))
            let hopSamples = Int(sampleRate * hopDuration)
            audioBuffer.removeFirst(hopSamples) // slide window

            mlQueue.async { [weak self] in
                self?.runInference(on: window)
            }
        }
    }

    // MARK: - ML Inference

    private func runInference(on samples: [Float]) {
        guard let model = emotionModel else { return }

        do {
            // Compute mel spectrogram using Accelerate
            let melSpec = computeMelSpectrogram(samples)

            // Create MLMultiArray [1, melBins, timeFrames]
            let timeFrames = melSpec.count / melBins
            let input = try MLMultiArray(
                shape: [1, melBins, timeFrames] as [NSNumber],
                dataType: .float32
            )
            for i in 0..<melSpec.count {
                input[i] = NSNumber(value: melSpec[i])
            }

            let featureProvider = try MLDictionaryFeatureProvider(
                dictionary: ["audio_features": MLFeatureValue(multiArray: input)]
            )

            let result = try model.prediction(from: featureProvider)

            if let probs = result.featureValue(
                for: "emotionProbabilities"
            )?.dictionaryValue as? [String: NSNumber] {
                let emotions = probs.mapValues { $0.doubleValue }

                DispatchQueue.main.async { [weak self] in
                    self?.currentEmotions = emotions
                    self?.dominantEmotion = emotions
                        .max(by: { $0.value < $1.value })?.key ?? "neutral"
                }
            }
        } catch {
            print("Inference error: \(error)")
        }
    }

    // MARK: - Feature Extraction (simplified)

import SwiftUI

struct EmotionView: View {

                manager.currentEmotions.sorted(by: { $0.value > $1.value }),

            ) { emotion, score in
                HStack {
                    Text(emotion.capitalized)
                        .frame(width: 80, alignment: .leading)
                        .font(.caption)
                    ProgressView(value: score)
                        .tint(.indigo)
                    Text(String(format: "%.0f%%", score * 100))
                        .font(.caption.monospacedDigit())
                        .frame(width: 40)
                }
            }

            Button(manager.isAnalyzing ? "Stop" : "Start") {
                if manager.isAnalyzing {
                    manager.stop()
                } else {
                    try? manager.start()
                }
            }
            .buttonStyle(.borderedProminent)
        }
        .padding()

Recommended Architecture for a Swift Transcription App

Our Recommendation
  1. Primary: On-device Core ML model (distilled or Create ML-trained) for real-time acoustic emotion detection during transcription
  2. Enhancement: Apple Foundation Models framework for semantic emotion analysis of transcribed text — free, private, no API key
  3. Preprocessing: FluidAudio or custom VAD to segment speech and identify speakers
  4. Optional: Cloud API (Hume AI) for detailed post-hoc analysis of completed recordings, with user consent

Privacy & Compliance

On-device emotion detection is a significant privacy advantage. Here is how it maps to major regulatory frameworks.

RegulationOn-Device ImpactCloud Impact
GDPR No cross-border transfer. No data processor agreements needed. Minimal breach surface. Requires DPA with cloud provider. Cross-border transfer rules apply. Must handle data deletion requests.
EU AI Act Fewer restrictions for on-device processing. Emotion recognition banned in workplaces/schools regardless. Higher scrutiny. Must demonstrate compliance for high-risk classification.
HIPAA No PHI transmitted. Simplifies compliance for health-adjacent apps. Requires BAA with cloud provider. Encryption in transit mandatory.
User Trust "Your audio never leaves your device" — clear, verifiable privacy promise. Must explain what data is sent, stored, and for how long.
Important: EU AI Act
The EU AI Act bans emotion recognition in workplace and educational settings regardless of whether processing is on-device or cloud. If your app could be used in these contexts, consult legal counsel on compliance.
