SwiftUI Modal Patterns

SwiftUI Modal Navigation Patterns

A comprehensive guide to sheet, fullScreenCover, popover, alert, and confirmationDialog — covering presentation APIs, customization options, lifecycle management, and best practices for iOS 16+ and 17+.

Scope
This guide covers all five modal presentation types in SwiftUI, including the customization modifiers introduced in iOS 16 and 17. All code examples compile against Xcode 15+ targeting iOS 17 unless noted otherwise.

SwiftUI's Modal Presentation Model

SwiftUI presents modal content through view modifiers attached to the view hierarchy. Unlike UIKit, where you imperatively call present(_:animated:), SwiftUI uses a declarative, state-driven approach: you bind a piece of state to a presentation modifier, and SwiftUI handles showing and hiding the view when that state changes.

Every modal modifier follows one of two signatures:

  • Boolean binding — .sheet(isPresented: $showSheet) { ... }
  • Item binding — .sheet(item: $selectedItem) { item in ... }

The boolean pattern uses a simple Bool toggle. The item pattern uses an Optional<Identifiable> value — the sheet is shown when the value is non-nil, and dismissed when it becomes nil. Both patterns have an optional onDismiss callback that fires after the modal is fully dismissed.

Key Principles

  • Modals are view modifiers — they attach to a view in the hierarchy and present content over it.
  • Presentation is driven by state — mutating the bound state triggers transitions automatically.
  • Each view can have one active sheet modifier at a time — multiple .sheet() modifiers on the same view produces undefined behavior (the last one wins).
  • Modals create a new environment scope — presented views get their own environment, which is why @Environment(\.dismiss) works from within the modal.
  • Sheets, fullScreenCovers, and popovers present view hierarchies. Alerts and confirmationDialogs present system-managed UI.

Presentation Lifecycle

Understanding the lifecycle of a modal presentation helps with managing state transitions and cleanup. SwiftUI handles animation and state synchronization automatically, but the onDismiss callback fires after the animation completes.

Figure 2 — Modal presentation lifecycle state diagram
Important Timing
The onDismiss callback fires after the dismiss animation completes and the state has been reset. This is the safe place to trigger follow-up presentations or clean up resources. Do not set state for the next modal in the same frame as dismissal — use onDismiss or DispatchQueue.main.async.

Sheet

The .sheet() modifier presents a view as a card that slides up from the bottom. On iPhone, it covers most of the screen by default but leaves a small portion of the parent visible at the top. On iPad and Mac, it appears as a centered card. Starting with iOS 16, you can control the height with presentation detents.

sheet(isPresented:) — Boolean Pattern

struct SheetBooleanExample: View {
    @State private var showSettings = false

    var body: some View {
        NavigationStack {
            List {
                Text("Main Content")
            }
            .navigationTitle("Home")
            .toolbar {
                Button("Settings") {
                    showSettings = true
                }
            }
            .sheet(isPresented: $showSettings) {
                SettingsView()
            }
        }
    }
}

struct SettingsView: View {
    @Environment(\.dismiss) private var dismiss

    var body: some View {
        NavigationStack {
            Form {
                Section("Preferences") {
                    Toggle("Notifications", isOn: .constant(true))
                    Toggle("Dark Mode", isOn: .constant(false))
                }
            }
            .navigationTitle("Settings")
            .navigationBarTitleDisplayMode(.inline)
            .toolbar {
                ToolbarItem(placement: .confirmationAction) {
                    Button("Done") { dismiss() }
                }
            }
        }
    }
}

sheet(item:) — Item-Based Pattern (Recommended)

struct User: Identifiable {
    let id = UUID()
    let name: String
    let email: String
}

struct SheetItemExample: View {
    @State private var selectedUser: User?

    let users = [
        User(name: "Alice", email: "alice@example.com"),
        User(name: "Bob", email: "bob@example.com"),
        User(name: "Charlie", email: "charlie@example.com")
    ]

    var body: some View {
        NavigationStack {
            List(users) { user in
                Button(user.name) {
                    selectedUser = user
                }
            }
            .navigationTitle("Users")
            .sheet(item: $selectedUser) { user in
                // 'user' is non-optional here -- safe to use directly
                UserDetailSheet(user: user)
            }
        }
    }
}

struct UserDetailSheet: View {
    let user: User
    @Environment(\.dismiss) private var dismiss

    var body: some View {
        VStack(spacing: 20) {
            Text(user.name)
                .font(.title.bold())
            Text(user.email)
                .foregroundStyle(.secondary)
            Button("Close") { dismiss() }
                .buttonStyle(.borderedProminent)
        }
        .padding()
    }
}
Best Practice
Prefer the item-based pattern over the boolean pattern. It eliminates the class of bugs where the sheet content does not match the intended data, reduces the number of state properties, and gives you a non-optional value in the closure.

Full Screen Cover

.fullScreenCover() presents a view that covers the entire screen. Unlike sheets, there is no swipe-to-dismiss gesture, so you must provide an explicit way for the user to dismiss the view. Use this for immersive flows like onboarding, authentication, media players, or multi-step wizards.

struct FullScreenCoverExample: View {
    @State private var showOnboarding = false

    var body: some View {
        VStack(spacing: 20) {
            Text("Welcome back!")
                .font(.largeTitle.bold())
            Button("Show Onboarding") {
                showOnboarding = true
            }
            .buttonStyle(.borderedProminent)
        }
        .fullScreenCover(isPresented: $showOnboarding) {
            OnboardingView()
        }
    }
}

struct OnboardingView: View {
    @Environment(\.dismiss) private var dismiss
    @State private var currentPage = 0

    let pages = [
        ("Welcome", "Get started with our app"),
        ("Features", "Discover powerful tools"),
        ("Ready", "You're all set!")
    ]

    var body: some View {
        VStack(spacing: 24) {
            Spacer()

            Text(pages[currentPage].0)
                .font(.largeTitle.bold())
            Text(pages[currentPage].1)
                .font(.title3)
                .foregroundStyle(.secondary)

            Spacer()

            HStack(spacing: 8) {
                ForEach(0..<pages.count, id: \.self) { i in
                    Circle()
                        .fill(i == currentPage ? Color.accentColor : Color.gray.opacity(0.3))
                        .frame(width: 8, height: 8)
                }
            }

            if currentPage < pages.count - 1 {
                Button("Next") {
                    withAnimation { currentPage += 1 }
                }
                .buttonStyle(.borderedProminent)
                .controlSize(.large)
            } else {
                Button("Get Started") {
                    dismiss()
                }
                .buttonStyle(.borderedProminent)
                .controlSize(.large)
            }

            Button("Skip") { dismiss() }
                .foregroundStyle(.secondary)
                .padding(.bottom, 40)
        }
        .padding()
        .interactiveDismissDisabled()
    }
}

fullScreenCover with item binding

enum FullScreenDestination: Identifiable {
    case login
    case player(videoURL: URL)
    case imageViewer(image: String)

    var id: String {
        switch self {
        case .login: return "login"
        case .player(let url): return "player-\(url.absoluteString)"
        case .imageViewer(let img): return "image-\(img)"
        }
    }
}

struct ContentView: View {
    @State private var activeDestination: FullScreenDestination?

    var body: some View {
        VStack(spacing: 16) {
            Button("Sign In") {
                activeDestination = .login
            }
            Button("Play Video") {
                activeDestination = .player(
                    videoURL: URL(string: "https://example.com/video.mp4")!
                )
            }
        }
        .fullScreenCover(item: $activeDestination) { destination in
            switch destination {
            case .login:
                LoginView()
            case .player(let url):
                VideoPlayerView(url: url)
            case .imageViewer(let image):
                ImageViewerView(imageName: image)
            }
        }
    }
}

Popover

Popovers display a floating view anchored to a source view with an arrow. On iPad and Mac, they appear as a floating bubble. On iPhone, SwiftUI automatically adapts popovers to sheets. Use popovers for contextual information, quick actions, or supplementary details.

struct PopoverExample: View {
    @State private var showPopover = false

    var body: some View {
        VStack {
            Button("Show Info") {
                showPopover.toggle()
            }
            .popover(isPresented: $showPopover,
                     attachmentAnchor: .point(.bottom),
                     arrowEdge: .top) {
                VStack(alignment: .leading, spacing: 12) {
                    Text("Quick Info")
                        .font(.headline)
                    Text("This popover appears anchored to the button. On iPad it floats; on iPhone it becomes a sheet.")
                        .font(.subheadline)
                        .foregroundStyle(.secondary)
                    Link("Learn More",
                         destination: URL(string: "https://developer.apple.com")!)
                }
                .padding()
                .frame(width: 280)
            }
        }
    }
}

Popover with Item Binding

struct InfoItem: Identifiable {
    let id = UUID()
    let title: String
    let detail: String
}

struct PopoverItemExample: View {
    @State private var activeInfo: InfoItem?

    var body: some View {
        HStack(spacing: 20) {
            Button("Pricing") {
                activeInfo = InfoItem(
                    title: "Pricing",
                    detail: "Starting at $9.99/month"
                )
            }
            Button("Support") {
                activeInfo = InfoItem(
                    title: "Support",
                    detail: "24/7 email and chat"
                )
            }
        }
        .popover(item: $activeInfo,
                 attachmentAnchor: .point(.bottom),
                 arrowEdge: .top) { info in
            VStack(alignment: .leading, spacing: 8) {
                Text(info.title).font(.headline)
                Text(info.detail).font(.body)
            }
            .padding()
            .frame(minWidth: 200)
        }
    }
}
Platform Differences
The arrowEdge parameter is only respected on macOS. On iPadOS, the system may ignore it and place the popover where it fits best. On iPhone, popovers become sheets automatically.

Attachment Anchors

The attachmentAnchor parameter accepts .point(UnitPoint) or .rect(Anchor<CGRect>.Source) values. Common anchor points:

  • .point(.top) — top center of the source view
  • .point(.bottom) — bottom center
  • .point(.leading) / .point(.trailing)
  • .point(.topLeading), .point(.bottomTrailing), etc.
  • .rect(.bounds) — anchored to the full bounds of the view

Alert

Alerts present a system-managed dialog centered on screen. They are ideal for critical decisions that require the user's attention: confirming destructive actions, displaying error messages, or asking for confirmation. Alert content is limited to a title, optional message, and buttons.

struct AlertExample: View {
    @State private var showDeleteAlert = false
    @State private var itemToDelete: String?

    var body: some View {
        List {
            ForEach(["Document A", "Document B", "Document C"], id: \.self) { doc in
                HStack {
                    Text(doc)
                    Spacer()
                    Button("Delete", role: .destructive) {
                        itemToDelete = doc
                        showDeleteAlert = true
                    }
                }
            }
        }
        .alert("Delete Document",
               isPresented: $showDeleteAlert,
               presenting: itemToDelete) { item in
            Button("Delete \"\(item)\"", role: .destructive) {
                // Perform deletion
                print("Deleted: \(item)")
            }
            Button("Cancel", role: .cancel) { }
        } message: { item in
            Text("Are you sure you want to delete \"\(item)\"? This action cannot be undone.")
        }
    }
}

Alert with Text Field (iOS 16+)

struct AlertTextFieldExample: View {
    @State private var showRenameAlert = false
    @State private var newName = ""

    var body: some View {
        Button("Rename Item") {
            newName = ""
            showRenameAlert = true
        }
        .alert("Rename", isPresented: $showRenameAlert) {
            TextField("New name", text: $newName)
            Button("Rename") {
                print("Renamed to: \(newName)")
            }
            Button("Cancel", role: .cancel) { }
        } message: {
            Text("Enter a new name for this item.")
        }
    }
}

Error Alert Pattern

struct ErrorAlertExample: View {
    @State private var appError: AppError?

    var body: some View {
        Button("Trigger Error") {
            appError = AppError(
                title: "Network Error",
                message: "Unable to reach the server. Check your connection."
            )
        }
        .alert(appError?.title ?? "Error",
               isPresented: Binding(
                   get: { appError != nil },
                   set: { if !$0 { appError = nil } }
               )) {
            Button("Retry") {
                // retry logic
            }
            Button("OK", role: .cancel) { }
        } message: {
            Text(appError?.message ?? "")
        }
    }
}

struct AppError {
    let title: String
    let message: String
}

Confirmation Dialog

.confirmationDialog() presents an action-sheet-style interface that slides up from the bottom on iPhone, or appears as a popover on iPad. It is designed for presenting multiple options or actions. Unlike alerts, it supports many buttons and can be dismissed by tapping outside.

struct ConfirmationDialogExample: View {
    @State private var showSortOptions = false
    @State private var sortOrder = "Name"

    var body: some View {
        NavigationStack {
            List {
                Text("Sorted by: \(sortOrder)")
            }
            .navigationTitle("Items")
            .toolbar {
                Button("Sort") {
                    showSortOptions = true
                }
            }
            .confirmationDialog("Sort By",
                                isPresented: $showSortOptions,
                                titleVisibility: .visible) {
                Button("Name") { sortOrder = "Name" }
                Button("Date Created") { sortOrder = "Date Created" }
                Button("Date Modified") { sortOrder = "Date Modified" }
                Button("Size") { sortOrder = "Size" }
                Button("Cancel", role: .cancel) { }
            } message: {
                Text("Choose how to sort your items.")
            }
        }
    }
}

Destructive Confirmation Dialog

struct DestructiveConfirmationExample: View {
    @State private var showDeleteOptions = false
    @State private var photos: [String] = ["IMG_001", "IMG_002", "IMG_003"]

    var body: some View {
        List {
            ForEach(photos, id: \.self) { photo in
                Text(photo)
            }
        }
        .toolbar {
            Button("Clear") { showDeleteOptions = true }
        }
        .confirmationDialog(
            "Remove Photos",
            isPresented: $showDeleteOptions,
            titleVisibility: .visible
        ) {
            Button("Delete All", role: .destructive) {
                photos.removeAll()
            }
            Button("Delete Oldest") {
                photos.removeFirst()
            }
            Button("Cancel", role: .cancel) { }
        } message: {
            Text("This will permanently remove the selected photos.")
        }
    }
}
Title Visibility
By default, confirmationDialog hides the title on iPhone. Pass titleVisibility: .visible to show it. The title is always shown on iPad/Mac where the dialog appears as a popover.

Boolean vs Item-Based Presentation

SwiftUI offers two binding patterns for every modal modifier. Understanding when to use each is crucial for clean, bug-free code.

Boolean Pattern

// Simple, works for single-purpose sheets
@State private var showProfile = false

.sheet(isPresented: $showProfile) {
    ProfileView()
}

Item-Based Pattern

// Scales better, passes data safely
@State private var selectedItem: Item?

.sheet(item: $selectedItem) { item in
    // 'item' is non-optional and guaranteed valid
    ItemDetailView(item: item)
}
AspectBoolean PatternItem Pattern
State propertiesOne Bool per sheetOne Optional per sheet
Data passingSeparate state + risk of mismatchBuilt-in, type-safe
Multiple sheetsMultiple Bools (messy)Single enum (clean)
Race conditionsPossible if data changes before sheet appearsCaptured at presentation time
Dismiss behaviorSet Bool to falseSet Optional to nil
Best forStatic sheets (settings, about)Data-driven sheets (detail, edit)

Enum-Based Pattern for Multiple Sheets

enum SheetType: Identifiable {
    case profile
    case settings
    case editItem(Item)

    var id: String {
        switch self {
        case .profile: return "profile"
        case .settings: return "settings"
        case .editItem(let item): return "edit-\(item.id)"
        }
    }
}

struct MultiSheetView: View {
    @State private var activeSheet: SheetType?

    var body: some View {
        VStack(spacing: 16) {
            Button("Profile") { activeSheet = .profile }
            Button("Settings") { activeSheet = .settings }
            Button("Edit") {
                activeSheet = .editItem(
                    Item(id: "1", name: "Example")
                )
            }
        }
        .sheet(item: $activeSheet) { sheet in
            switch sheet {
            case .profile:
                ProfileView()
            case .settings:
                SettingsView()
            case .editItem(let item):
                EditItemView(item: item)
            }
        }
    }
}
Recommendation
Use the enum-based item pattern as your default approach. It scales from one sheet to many without refactoring, prevents data-binding race conditions, and keeps all sheet destinations visible in a single type.

Dismiss Patterns

SwiftUI provides several ways to dismiss a modal presentation. The right choice depends on your architecture and where the dismiss logic lives.

@Environment(\.dismiss) — Standard Approach

struct ModalContent: View {
    @Environment(\.dismiss) private var dismiss

    var body: some View {
        VStack {
            Text("Modal Content")
            Button("Close") {
                dismiss()
            }
        }
    }
}

Binding-Based Dismiss

// Parent controls presentation
struct ParentView: View {
    @State private var showModal = false

    var body: some View {
        Button("Show") { showModal = true }
            .sheet(isPresented: $showModal) {
                ChildView(isPresented: $showModal)
            }
    }
}

// Child can dismiss by setting binding to false
struct ChildView: View {
    @Binding var isPresented: Bool

    var body: some View {
        Button("Done") {
            isPresented = false
        }
    }
}

Item-Based Dismiss (set to nil)

struct ItemDismissExample: View {
    @State private var selectedItem: Item?

    var body: some View {
        Button("Show") {
            selectedItem = Item(id: "1", name: "Test")
        }
        .sheet(item: $selectedItem) { item in
            VStack {
                Text(item.name)
                Button("Close") {
                    selectedItem = nil  // Dismisses the sheet
                }
            }
        }
    }
}
Prefer @Environment(\.dismiss)
The environment-based dismiss action is the most flexible approach. It works in any presented context (sheet, fullScreenCover, NavigationStack) without the child needing to know how it was presented.

Passing Data Back on Dismiss

A common need is for the modal to send data back to the presenting view when it dismisses. Here are the main approaches.

Callback Closure

struct AddItemSheet: View {
    @Environment(\.dismiss) private var dismiss
    @State private var itemName = ""
    let onSave: (String) -> Void

    var body: some View {
        NavigationStack {
            Form {
                TextField("Item name", text: $itemName)
            }
            .navigationTitle("New Item")
            .toolbar {
                ToolbarItem(placement: .cancellationAction) {
                    Button("Cancel") { dismiss() }
                }
                ToolbarItem(placement: .confirmationAction) {
                    Button("Save") {
                        onSave(itemName)
                        dismiss()
                    }
                    .disabled(itemName.isEmpty)
                }
            }
        }
    }
}

struct ParentWithCallback: View {
    @State private var showAddItem = false
    @State private var items: [String] = []

    var body: some View {
        List(items, id: \.self) { Text($0) }
            .toolbar {
                Button("Add") { showAddItem = true }
            }
            .sheet(isPresented: $showAddItem) {
                AddItemSheet { newItem in
                    items.append(newItem)
                }
            }
    }
}

Binding Pattern

struct ColorPickerSheet: View {
    @Environment(\.dismiss) private var dismiss
    @Binding var selectedColor: Color

    let colors: [Color] = [.red, .blue, .green, .orange, .purple]

    var body: some View {
        VStack(spacing: 16) {
            Text("Pick a Color").font(.headline)
            HStack(spacing: 12) {
                ForEach(colors, id: \.self) { color in
                    Circle()
                        .fill(color)
                        .frame(width: 44, height: 44)
                        .overlay(
                            Circle().stroke(.white, lineWidth:
                                selectedColor == color ? 3 : 0)
                        )
                        .onTapGesture {
                            selectedColor = color
                            dismiss()
                        }
                }
            }
        }
        .padding()
    }
}

struct ParentWithBinding: View {
    @State private var showPicker = false
    @State private var currentColor: Color = .blue

    var body: some View {
        VStack {
            Circle()
                .fill(currentColor)
                .frame(width: 100, height: 100)
            Button("Change Color") { showPicker = true }
        }
        .sheet(isPresented: $showPicker) {
            ColorPickerSheet(selectedColor: $currentColor)
                .presentationDetents([.height(200)])
        }
    }
}

onDismiss Callback

struct OnDismissExample: View {
    @State private var showEditor = false
    @State private var draftText = ""

    var body: some View {
        Text("Last draft: \(draftText)")
            .sheet(isPresented: $showEditor, onDismiss: {
                // Fires after sheet is fully dismissed
                print("Editor dismissed. Draft: \(draftText)")
                saveDraft()
            }) {
                TextEditor(text: $draftText)
                    .padding()
            }
    }

    func saveDraft() {
        // Save logic
    }
}

Multiple Sheets from One View

Presenting multiple different sheets from the same view is a common need but has important rules. The single enum pattern is the cleanest solution.

Do Not: Multiple .sheet() Modifiers on the Same View
Attaching more than one .sheet() modifier to the same view is undefined behavior. Only the last modifier will work, and Xcode may log a warning: "Currently, only presenting a single sheet is supported."

Wrong: Multiple sheet modifiers

// DO NOT DO THIS -- only the last .sheet() will work
struct BrokenMultiSheet: View {
    @State private var showA = false
    @State private var showB = false

    var body: some View {
        VStack {
            Button("Sheet A") { showA = true }
            Button("Sheet B") { showB = true }
        }
        .sheet(isPresented: $showA) { Text("A") }  // IGNORED
        .sheet(isPresented: $showB) { Text("B") }  // Only this works
    }
}

Correct: Enum-based single sheet modifier

enum ActiveSheet: Identifiable {
    case compose
    case settings
    case share

    var id: Self { self }
}

struct CorrectMultiSheet: View {
    @State private var activeSheet: ActiveSheet?

    var body: some View {
        VStack(spacing: 16) {
            Button("Compose") { activeSheet = .compose }
            Button("Settings") { activeSheet = .settings }
            Button("Share") { activeSheet = .share }
        }
        .sheet(item: $activeSheet) { sheet in
            switch sheet {
            case .compose:  ComposeView()
            case .settings: SettingsView()
            case .share:    ShareView()
            }
        }
    }
}

Alternative: Sheet modifiers on different child views

// Each button owns its own sheet -- this works
struct SplitSheetOwnership: View {
    var body: some View {
        VStack(spacing: 16) {
            SheetButtonA()
            SheetButtonB()
        }
    }
}

struct SheetButtonA: View {
    @State private var showSheet = false
    var body: some View {
        Button("Sheet A") { showSheet = true }
            .sheet(isPresented: $showSheet) {
                Text("Content A")
            }
    }
}

struct SheetButtonB: View {
    @State private var showSheet = false
    var body: some View {
        Button("Sheet B") { showSheet = true }
            .sheet(isPresented: $showSheet) {
                Text("Content B")
            }
    }
}

Chaining Modals

Sometimes you need to dismiss one modal and immediately present another. This requires careful timing because SwiftUI needs to complete the dismiss animation before starting a new presentation.

struct ChainingModalsExample: View {
    @State private var showFirstSheet = false
    @State private var showSecondSheet = false

    var body: some View {
        Button("Start Flow") {
            showFirstSheet = true
        }
        .sheet(isPresented: $showFirstSheet, onDismiss: {
            // Present second sheet AFTER first is fully dismissed
            showSecondSheet = true
        }) {
            VStack {
                Text("Step 1")
                    .font(.title)
                Button("Continue to Step 2") {
                    showFirstSheet = false
                }
                .buttonStyle(.borderedProminent)
            }
            .padding()
        }
        .sheet(isPresented: $showSecondSheet) {
            VStack {
                Text("Step 2")
                    .font(.title)
                Text("Second sheet presented after first dismissed.")
            }
            .padding()
        }
    }
}
Timing is Critical
The onDismiss callback fires after the dismiss animation completes. This is the only reliable place to trigger the next modal. Do not try to dismiss and present in the same action — the second presentation will be lost.

Known Issue: Title Display Mode in Sheets
On iOS 17, sheets using NavigationStack initialize with .inline title display mode. If the user partially swipes down and releases, the title may switch to .large style. Use .navigationBarTitleDisplayMode(.inline) explicitly to prevent layout jumps.

Disable Interactive Dismiss iOS 15+

By default, users can swipe down to dismiss a sheet. For forms with unsaved data or critical flows, you may want to prevent this.

struct InteractiveDismissExample: View {
    @State private var showEditor = false

    var body: some View {
        Button("Edit Document") { showEditor = true }
            .sheet(isPresented: $showEditor) {
                DocumentEditor()
            }
    }
}

struct DocumentEditor: View {
    @Environment(\.dismiss) private var dismiss
    @State private var text = ""
    @State private var showDiscardAlert = false

    private var hasChanges: Bool { !text.isEmpty }

    var body: some View {
        NavigationStack {
            TextEditor(text: $text)
                .padding()
                .navigationTitle("Edit")
                .navigationBarTitleDisplayMode(.inline)
                .toolbar {
                    ToolbarItem(placement: .cancellationAction) {
                        Button("Cancel") {
                            if hasChanges {
                                showDiscardAlert = true
                            } else {
                                dismiss()
                            }
                        }
                    }
                    ToolbarItem(placement: .confirmationAction) {
                        Button("Save") {
                            // Save logic
                            dismiss()
                        }
                    }
                }
                // Prevent swipe-to-dismiss when there are unsaved changes
                .interactiveDismissDisabled(hasChanges)
                .alert("Discard Changes?",
                       isPresented: $showDiscardAlert) {
                    Button("Discard", role: .destructive) {
                        dismiss()
                    }
                    Button("Keep Editing", role: .cancel) { }
                } message: {
                    Text("You have unsaved changes.")
                }
        }
    }
}
Conditional Disable
The modifier accepts a Bool parameter. Pass a computed property like hasChanges to conditionally disable interactive dismiss only when the user has made edits.

Rules Summary

  1. One sheet modifier per view level. Multiple .sheet() modifiers on the same view is undefined behavior. Use the enum pattern or move modifiers to child views.
  2. Sheets can present sheets. A sheet can have its own .sheet() modifier, creating a visual stack. The system handles layering automatically.
  3. Sequential presentation requires onDismiss. To dismiss one sheet and show another, use the onDismiss callback of the first sheet to trigger the second.
  4. Alerts and confirmationDialogs coexist with sheets. You can have a .sheet() and an .alert() on the same view. They use different presentation channels.
  5. fullScreenCover and sheet cannot coexist on the same view. They share the same presentation channel. Use child views or the enum pattern.

Common Pitfalls

1. Multiple .sheet() Modifiers on the Same View

Problem: Only the last .sheet() modifier takes effect.

Fix: Use a single .sheet(item:) with an enum, or attach sheets to different child views.

2. Boolean Binding Data Mismatch

Problem: With isPresented, you set a separate @State variable for data, but the sheet reads old data on first presentation because the state update and boolean toggle happen in the same render cycle.

Fix: Use the item-based pattern. The item is captured at presentation time, guaranteeing the correct value.

3. Attempting to Present Multiple Modals Simultaneously

Problem: Setting two boolean states to true in the same action to show two sheets. Only one appears.

Fix: SwiftUI supports one sheet at a time per presentation channel. Use onDismiss to sequence them, or nest sheets inside sheets.

4. Missing Dismiss Button on fullScreenCover

Problem: Unlike sheets, full-screen covers have no built-in dismiss gesture. Users get stuck.

Fix: Always provide a visible close or done button when using fullScreenCover.

5. NavigationStack Title Glitch in Sheets

Problem: On iOS 17, the navigation title in a sheet may switch between inline and large after a partial swipe.

Fix: Explicitly set .navigationBarTitleDisplayMode(.inline) on your root view inside the sheet.

6. Popover Becoming a Sheet on iPhone

Problem: Popovers automatically adapt to sheets on compact-width devices (iPhone).

Fix: Use .presentationCompactAdaptation(.none) (iOS 16.4+) to force popover style, or design your content to work as both a popover and a sheet.

7. Dismissing and Immediately Presenting

Problem: Setting showSheetA = false; showSheetB = true in the same action. The second sheet may not appear.

Fix: Use onDismiss on the first sheet to trigger the second presentation after the animation completes.

8. Using @State for Sheet Content in Lists

Problem: In a List, setting @State and toggling a boolean for the sheet causes the wrong item to appear because the state updates after the sheet content closure is evaluated.

Fix: Use .sheet(item:) with the list item directly. The closure receives the correct item as a non-optional parameter.
