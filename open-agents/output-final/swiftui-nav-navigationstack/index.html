NavigationStack & NavigationPath iOS 16+

A comprehensive reference guide to SwiftUI's modern stack-based navigation system. Covers the declarative navigation model, programmatic control, type-erased paths, state persistence, and complete Xcode-ready code examples.

Overview

Starting with iOS 16 (WWDC 2022), Apple introduced NavigationStack as the successor to NavigationView. It provides a data-driven, stack-based navigation model where the navigation state is represented by a collection of values, and the framework resolves those values to views using .navigationDestination(for:) modifiers.

The key innovation is separating what you want to navigate to (a value) from how it should be displayed (a destination view). This separation enables programmatic navigation, deep linking, state restoration, and testable navigation logic -- all of which were cumbersome or impossible with the old NavigationView / NavigationLink(destination:) API.

Key Components
  • NavigationStack -- the container that manages a stack of views
  • NavigationLink(value:) -- pushes a Hashable value onto the stack
  • .navigationDestination(for:) -- resolves a value type to a destination view
  • NavigationPath -- a type-erased collection for heterogeneous navigation state

How NavigationStack Replaced NavigationView

Prior to iOS 16, SwiftUI developers used NavigationView with view-based NavigationLink(destination:) modifiers. While functional for simple use cases, this approach had significant limitations:

Aspect NavigationView (deprecated) NavigationStack (iOS 16+)
Navigation model View-based (destination is a view) Value-based (destination resolved from data)
Programmatic push Hacky (isActive bindings, tag/selection) First-class (append to path)
Pop to root Not directly supported path.removeAll() or path = []
Pop to specific level Not supported path.removeLast(n)
Deep linking Complex, fragile Set path to desired sequence
State serialization Not supported Built-in Codable support
Lazy loading Destination views eagerly initialized Destinations initialized only when pushed
Column navigation NavigationView with columns Use NavigationSplitView instead
Deprecation Notice

NavigationView is officially deprecated as of iOS 16. Apple's migration guide recommends replacing single-column NavigationView with NavigationStack, and multi-column NavigationView with NavigationSplitView. Existing apps using NavigationView will continue to compile but won't receive new navigation features.

The Declarative Navigation Model

SwiftUI's navigation model follows the same principle as the rest of the framework: state drives UI. Your navigation stack is a function of data. When you push a value onto the path, SwiftUI finds the matching .navigationDestination(for:) modifier and renders the corresponding view. When you remove a value, that view is popped.

This model means:

  • The path array is the single source of truth for what's on the stack
  • Navigation is testable -- you can assert against the path contents without rendering views
  • Navigation is serializable -- save the path to disk, restore it on next launch
  • Navigation is decoupled -- the link doesn't need to know what view it opens

Basic Usage

Simple NavigationStack with NavigationLink

The simplest form of NavigationStack doesn't even require a path binding. It works just like the old NavigationView but with lazy destination loading and the modern API:

Swift 
import SwiftUI

struct SimpleNavigationExample: View {
    var body: some View {
        NavigationStack {
            List {
                NavigationLink("Go to Settings") {
                    SettingsView()
                }
                NavigationLink("Go to Profile") {
                    ProfileView()
                }
            }
            .navigationTitle("Home")
        }
    }
}

struct SettingsView: View {
    var body: some View {
        Text("Settings Screen")
            .navigationTitle("Settings")
    }
}

struct ProfileView: View {
    var body: some View {
        Text("Profile Screen")
            .navigationTitle("Profile")
    }
}
Note

This "view-destination" style still works but doesn't give you programmatic control. For real-world apps, prefer the value-based pattern shown below.

Value-Based NavigationLink

The modern approach uses NavigationLink(value:) to push a Hashable value instead of a view. A separate .navigationDestination(for:) modifier maps that value to the destination view:

Swift 
import SwiftUI

// 1. Define a Hashable model for navigation
struct Recipe: Identifiable, Hashable {
    let id = UUID()
    let name: String
    let prepTime: Int // minutes
}

struct RecipeListView: View {
    let recipes = [
        Recipe(name: "Pasta Carbonara", prepTime: 25),
        Recipe(name: "Caesar Salad", prepTime: 15),
        Recipe(name: "Chocolate Cake", prepTime: 60),
    ]

    var body: some View {
        NavigationStack {
            List(recipes) { recipe in
                // 2. Push a value, not a view
                NavigationLink(value: recipe) {
                    Text(recipe.name)
                }
            }
            // 3. Map the value type to a destination view
            .navigationDestination(for: Recipe.self) { recipe in
                RecipeDetailView(recipe: recipe)
            }
            .navigationTitle("Recipes")
        }
    }
}

struct RecipeDetailView: View {
    let recipe: Recipe

    var body: some View {
        VStack(alignment: .leading, spacing: 16) {
            Text(recipe.name)
                .font(.largeTitle)
            Text("Prep time: \(recipe.prepTime) minutes")
                .foregroundStyle(.secondary)
            Spacer()
        }
        .padding()
        .navigationTitle(recipe.name)
    }
}

The separation of concerns here is significant:

  • The NavigationLink only knows what data to push -- not what screen to show
  • The .navigationDestination only knows how to render a given data type -- not where the link is
  • The same .navigationDestination handles all links for that type, no matter where they appear in the view hierarchy

NavigationPath Deep Dive

How NavigationPath Works Internally

NavigationPath is a type-erased, heterogeneous collection that can hold values of any Hashable type. It behaves like an ordered stack where you can append values, remove from the end, and query the count -- but you cannot iterate over or index into its elements (they're type-erased).

Figure 2: NavigationPath state management -- a type-erased heterogeneous stack with limited public API

Key characteristics of NavigationPath:

  • Type erasure: Values are stored as any Hashable. You lose the ability to read them back with their original type.
  • Append-only access: You can append and remove, but you can't subscript (path[0] is not available).
  • Codable support: If all pushed values conform to Codable, the .codable property returns a CodableRepresentation for serialization. If any non-Codable value is in the path, it returns nil.
  • Swift 5.7 existentials: Under the hood, NavigationPath uses Swift's improved existential type handling to achieve ergonomic type erasure while retaining encoding capabilities.

Programmatic Navigation with NavigationPath

To control navigation programmatically, declare a @State property of type NavigationPath (or a typed array) and pass it as a binding to NavigationStack(path:):

Swift 
import SwiftUI

struct Product: Identifiable, Hashable {
    let id = UUID()
    let name: String
    let price: Double
}

struct ShopView: View {
    // The single source of truth for navigation state
    @State private var path = NavigationPath()

    let featuredProduct = Product(
        name: "AirPods Pro",
        price: 249.0
    )

    var body: some View {
        // Bind the path to NavigationStack
        NavigationStack(path: $path) {
            VStack(spacing: 20) {
                // Standard NavigationLink (also appends to path)
                NavigationLink(value: featuredProduct) {
                    Text("View Featured Product")
                }

                // Programmatic push via button
                Button("Jump to Checkout") {
                    // Push multiple values at once
                    path.append(featuredProduct)
                    path.append("checkout")
                }

                Text("Stack depth: \(path.count)")
                    .foregroundStyle(.secondary)
            }
            .navigationTitle("Shop")
            // Destination for Product values
            .navigationDestination(for: Product.self) { product in
                ProductDetailView(product: product, path: $path)
            }
            // Destination for String values
            .navigationDestination(for: String.self) { route in
                if route == "checkout" {
                    CheckoutView(path: $path)
                }
            }
        }
    }
}

struct ProductDetailView: View {
    let product: Product
    @Binding var path: NavigationPath

    var body: some View {
        VStack(spacing: 16) {
            Text(product.name).font(.title)
            Text("$\(product.price, specifier: "%.2f")")
                .font(.title2)
            Button("Proceed to Checkout") {
                path.append("checkout")
            }
        }
        .navigationTitle(product.name)
    }
}

struct CheckoutView: View {
    @Binding var path: NavigationPath

    var body: some View {
        VStack(spacing: 16) {
            Text("Checkout").font(.title)
            Button("Complete Purchase & Go Home") {
                // Pop everything -- return to root
                path.removeLast(path.count)
            }
        }
        .navigationTitle("Checkout")
    }
}

Pop to Root and Multi-Level Pop

Popping views is straightforward -- you just manipulate the path:

Swift 
// Pop one level (same as pressing the back button)
path.removeLast()

// Pop two levels
path.removeLast(2)

// Pop to root -- Option A: remove all items
path.removeLast(path.count)

// Pop to root -- Option B: replace with empty path
path = NavigationPath()

// Pop to root -- Option C (if using typed array [MyRoute])
// path.removeAll()
// path = []
UIKit Comparison

If you're coming from UIKit, think of path.removeLast(path.count) as the SwiftUI equivalent of navigationController?.popToRootViewController(animated: true). The key difference is that in SwiftUI you're modifying state rather than calling an imperative method -- SwiftUI handles the animation automatically.

Typed Arrays vs NavigationPath

You have two choices for the path binding: a typed array (homogeneous) or NavigationPath (heterogeneous). Each has distinct advantages.

Using a Typed Array

Swift 
import SwiftUI

enum AppRoute: Hashable {
    case productList
    case productDetail(id: UUID)
    case cart
    case checkout
    case confirmation(orderID: String)
}

struct TypedArrayExample: View {
    // Typed array -- all elements must be AppRoute
    @State private var path: [AppRoute] = []

    var body: some View {
        NavigationStack(path: $path) {
            VStack(spacing: 16) {
                Button("Browse Products") {
                    path.append(.productList)
                }
                Button("Go to Cart") {
                    path.append(.cart)
                }
                Button("Deep link: Product → Cart → Checkout") {
                    path = [
                        .productList,
                        .productDetail(id: UUID()),
                        .cart,
                        .checkout
                    ]
                }
            }
            .navigationTitle("Home")
            .navigationDestination(for: AppRoute.self) { route in
                switch route {
                case .productList:
                    Text("Product List").navigationTitle("Products")
                case .productDetail(let id):
                    Text("Product \(id)").navigationTitle("Detail")
                case .cart:
                    Text("Shopping Cart").navigationTitle("Cart")
                case .checkout:
                    Text("Checkout").navigationTitle("Checkout")
                case .confirmation(let orderID):
                    Text("Order \(orderID) confirmed!")
                        .navigationTitle("Confirmation")
                }
            }
        }
    }
}

Using NavigationPath (Heterogeneous)

Swift 
import SwiftUI

struct Category: Hashable {
    let name: String
}

struct Article: Hashable {
    let id: Int
    let title: String
}

struct NavigationPathExample: View {
    // NavigationPath can hold ANY Hashable type
    @State private var path = NavigationPath()

    var body: some View {
        NavigationStack(path: $path) {
            VStack(spacing: 16) {
                // Push a Category
                Button("Browse Tech") {
                    path.append(Category(name: "Technology"))
                }
                // Push an Article directly
                Button("Read Article #1") {
                    path.append(Article(id: 1, title: "SwiftUI Tips"))
                }
                // Push mixed types in sequence
                Button("Category → Article") {
                    path.append(Category(name: "Science"))
                    path.append(Article(id: 2, title: "Quantum Computing"))
                }
            }
            .navigationTitle("Home")
            .navigationDestination(for: Category.self) { category in
                Text("Category: \(category.name)")
                    .navigationTitle(category.name)
            }
            .navigationDestination(for: Article.self) { article in
                Text("Article: \(article.title)")
                    .navigationTitle(article.title)
            }
        }
    }
}

Comparison Table

Feature Typed Array [Route] NavigationPath
Heterogeneous types No -- single type only Yes -- any Hashable
Type safety Full compile-time safety Type-erased at runtime
Inspection Full array access (subscript, iterate, filter) Only count and isEmpty
Codable support Standard Codable (if Route: Codable) Via .codable property (returns nil if any value isn't Codable)
Enum routing Natural fit -- one enum covers all routes Not needed, but can still be used
Deep linking Set path = [.screen1, .screen2, ...] Append each value sequentially
Best for Apps with a single, well-defined route enum Apps with multiple independent model types pushed onto the stack
Recommendation

Prefer typed arrays with a route enum for most apps. You get full type safety, inspectability, and natural Codable conformance. Use NavigationPath when you genuinely need to push values of different, unrelated types -- for example, in a generic content browser where categories, articles, and users are all first-class navigation targets.

Combining Multiple .navigationDestination Modifiers

You can attach multiple .navigationDestination(for:) modifiers to handle different value types. SwiftUI matches the type of the pushed value to the appropriate destination closure:

Swift 
import SwiftUI

struct User: Hashable {
    let name: String
    let handle: String
}

struct Post: Hashable {
    let id: Int
    let content: String
}

struct MultiDestinationExample: View {
    @State private var path = NavigationPath()

    var body: some View {
        NavigationStack(path: $path) {
            List {
                Section("Users") {
                    NavigationLink(
                        value: User(name: "Alice", handle: "@alice")
                    ) {
                        Text("Alice")
                    }
                }
                Section("Posts") {
                    NavigationLink(
                        value: Post(id: 1, content: "Hello, world!")
                    ) {
                        Text("Post #1")
                    }
                }
                Section("Settings") {
                    NavigationLink(value: "settings") {
                        Text("App Settings")
                    }
                }
            }
            .navigationTitle("Feed")
            // Each type gets its own destination handler
            .navigationDestination(for: User.self) { user in
                VStack {
                    Text(user.name).font(.title)
                    Text(user.handle).foregroundStyle(.secondary)
                }
                .navigationTitle("Profile")
            }
            .navigationDestination(for: Post.self) { post in
                VStack(alignment: .leading) {
                    Text("Post #\(post.id)").font(.headline)
                    Text(post.content)
                }
                .padding()
                .navigationTitle("Post")
            }
            .navigationDestination(for: String.self) { route in
                switch route {
                case "settings":
                    Text("Settings Page").navigationTitle("Settings")
                default:
                    Text("Unknown route: \(route)")
                }
            }
        }
    }
}
Important Rules
  • Only one .navigationDestination(for:) per type is allowed in the same view hierarchy level. If you register two handlers for the same type, the one closest to the NavigationStack (highest in the view tree) wins.
  • If no matching destination is found for a pushed value type, SwiftUI will print a runtime warning: "A NavigationLink is presenting a value of type X but there is no matching navigationDestination declaration visible from the location of the link."
  • The .navigationDestination modifiers can be placed on any view inside the NavigationStack -- they don't have to be on the root view. However, placing them on the root is the most common and predictable pattern.

Passing Data Between Screens

Since navigation is value-driven, passing data is as simple as putting data in the value you push. The destination closure receives that value and can use it however it needs:

Swift 
import SwiftUI

// Rich data model -- contains everything the detail screen needs
struct Movie: Identifiable, Hashable {
    let id = UUID()
    let title: String
    let year: Int
    let director: String
    let rating: Double
}

struct MovieBrowserView: View {
    @State private var path = NavigationPath()

    let movies = [
        Movie(title: "Inception", year: 2010,
              director: "Christopher Nolan", rating: 8.8),
        Movie(title: "The Matrix", year: 1999,
              director: "The Wachowskis", rating: 8.7),
    ]

    var body: some View {
        NavigationStack(path: $path) {
            List(movies) { movie in
                // The entire Movie struct is the navigation value
                NavigationLink(value: movie) {
                    VStack(alignment: .leading) {
                        Text(movie.title).font(.headline)
                        Text("\(movie.year) • \(movie.director)")
                            .font(.caption)
                            .foregroundStyle(.secondary)
                    }
                }
            }
            .navigationTitle("Movies")
            .navigationDestination(for: Movie.self) { movie in
                MovieDetailScreen(movie: movie, path: $path)
            }
        }
    }
}

struct MovieDetailScreen: View {
    let movie: Movie
    @Binding var path: NavigationPath

    var body: some View {
        VStack(alignment: .leading, spacing: 12) {
            Text(movie.title)
                .font(.largeTitle)
                .fontWeight(.bold)
            HStack {
                Text("Directed by \(movie.director)")
                Spacer()
                Text("\(movie.year)")
            }
            .foregroundStyle(.secondary)
            Text("Rating: \(movie.rating, specifier: "%.1f")/10")
                .font(.title2)
            Spacer()
            Button("Back to All Movies") {
                path.removeLast(path.count)
            }
            .buttonStyle(.borderedProminent)
        }
        .padding()
        .navigationTitle(movie.title)
        .navigationBarTitleDisplayMode(.inline)
    }
}

Pattern: For simple cases, pass the whole model as the navigation value. For cases where the detail screen needs to fetch additional data, pass just an identifier (like a UUID or Int) and let the detail screen load the data from a store or network.

Persisting Navigation State

NavigationPath supports Codable serialization when all values in the path conform to Codable. This enables saving and restoring navigation state across app launches:

Swift 
import SwiftUI

@Observable
class NavigationStore {
    var path = NavigationPath() {
        didSet { save() }
    }

    private let savePath = URL.documentsDirectory
        .appending(path: "navigation.json")

    init() {
        // Attempt to restore saved navigation state
        if let data = try? Data(contentsOf: savePath),
           let representation = try? JSONDecoder().decode(
               NavigationPath.CodableRepresentation.self,
               from: data
           ) {
            path = NavigationPath(representation)
        }
    }

    private func save() {
        guard let representation = path.codable else {
            // Some values in path don't conform to Codable
            return
        }
        do {
            let data = try JSONEncoder().encode(representation)
            try data.write(to: savePath)
        } catch {
            print("Failed to save navigation: \(error)")
        }
    }
}

// Usage in your App
@main
struct MyApp: App {
    @State private var store = NavigationStore()

    var body: some Scene {
        WindowGroup {
            NavigationStack(path: $store.path) {
                HomeView()
                    .navigationDestination(for: String.self) { route in
                        Text("Route: \(route)")
                    }
            }
        }
    }
}
Caveat: Codable Stability

If your route types change between app versions (e.g., you rename an enum case or change associated values), previously saved navigation data may fail to decode. Always handle decoding failures gracefully by falling back to the root state. Consider versioning your route types if persistence is critical.

Router / Coordinator Pattern

For larger apps, centralizing navigation logic in a dedicated Router (sometimes called a Coordinator) keeps your views clean and your navigation testable:

Swift 
import SwiftUI

// MARK: - Route Definition
enum AppRoute: Hashable, Codable {
    case home
    case profile(userID: String)
    case settings
    case settingsDetail(SettingsSection)

    enum SettingsSection: String, Hashable, Codable {
        case account, notifications, privacy
    }
}

// MARK: - Router
@Observable
class AppRouter {
    var path: [AppRoute] = []

    func navigate(to route: AppRoute) {
        path.append(route)
    }

    func popToRoot() {
        path.removeAll()
    }

    func pop() {
        guard !path.isEmpty else { return }
        path.removeLast()
    }

    func pop(to route: AppRoute) {
        guard let index = path.lastIndex(of: route) else { return }
        path.removeLast(path.count - index - 1)
    }

    // Deep link handler
    func handleDeepLink(url: URL) {
        guard let components = URLComponents(url: url,
            resolvingAgainstBaseURL: false) else { return }

        path.removeAll()

        switch components.path {
        case let p where p.hasPrefix("/profile/"):
            let userID = String(p.dropFirst(9))
            path = [.profile(userID: userID)]
        case "/settings/notifications":
            path = [.settings, .settingsDetail(.notifications)]
        default:
            break
        }
    }
}

// MARK: - Destination Builder
struct RouteDestination: View {
    let route: AppRoute

    var body: some View {
        switch route {
        case .home:
            Text("Home")
        case .profile(let userID):
            Text("Profile: \(userID)").navigationTitle("Profile")
        case .settings:
            SettingsListView()
        case .settingsDetail(let section):
            Text("\(section.rawValue.capitalized) Settings")
                .navigationTitle(section.rawValue.capitalized)
        }
    }
}

// MARK: - Root View
struct ContentView: View {
    @State private var router = AppRouter()

    var body: some View {
        NavigationStack(path: $router.path) {
            List {
                NavigationLink(value: AppRoute.profile(userID: "alice")) {
                    Text("Alice's Profile")
                }
                NavigationLink(value: AppRoute.settings) {
                    Text("Settings")
                }
            }
            .navigationTitle("App")
            .navigationDestination(for: AppRoute.self) { route in
                RouteDestination(route: route)
            }
        }
        .environment(router)
        .onOpenURL { url in
            router.handleDeepLink(url: url)
        }
    }
}

struct SettingsListView: View {
    @Environment(AppRouter.self) private var router

    var body: some View {
        List {
            NavigationLink(value: AppRoute.settingsDetail(.account)) {
                Text("Account")
            }
            NavigationLink(value: AppRoute.settingsDetail(.notifications)) {
                Text("Notifications")
            }
            NavigationLink(value: AppRoute.settingsDetail(.privacy)) {
                Text("Privacy")
            }
        }
        .navigationTitle("Settings")
    }
}

State restoration Built-in Codable support Must build custom solution
Learning curve Moderate -- new mental model Low -- familiar push/pop
Minimum target iOS 16 / macOS 13 iOS 13 / macOS 10.15
Destination laziness Lazy -- created only when pushed Eager -- created at init time
Programmatic control Full -- push, pop, pop to root, pop to level Limited -- isActive or tag/selection
Multi-column / iPad Use NavigationSplitView separately NavigationView with .columns style
Complexity at scale Scales well with router pattern Becomes unwieldy with many screens

NavigationStack + Path vs Sheet/FullScreenCover

Pattern Best For Navigation Model
NavigationStack Drill-down flows (list → detail → sub-detail) Stack-based, linear
.sheet / .fullScreenCover Modal flows (creation forms, settings, alerts) Modal overlay
NavigationSplitView Multi-column layouts (sidebar → content → detail) Column-based, adaptive
TabView Top-level app sections Tab-based, parallel

When to Use This Approach

Use NavigationStack + Path When
  • You need programmatic navigation (push/pop from code, not just links)
  • Your app has deep linking requirements
  • You want to save and restore navigation state across sessions
  • You're building a drill-down flow (master → detail → sub-detail)
  • You want testable navigation logic
  • You're targeting iOS 16+ (which you should be for new projects in 2024+)
  • You need pop-to-root or pop-to-specific-screen functionality
Consider Alternatives When
  • Multi-column layout: Use NavigationSplitView for sidebar + content + detail on iPad
  • Modal flows: Use .sheet or .fullScreenCover for forms, settings, or creation flows
  • Tab-based navigation: Use TabView for top-level section switching
  • Simple two-screen app: Plain NavigationStack without a path binding may be sufficient
  • Supporting iOS 15 and below: You'll need NavigationView (consider a compatibility shim)

Common Pitfalls

1. Duplicate .navigationDestination for the same type

If you register two .navigationDestination(for: String.self) modifiers at the same level, the one closer to the NavigationStack root wins. The other is silently ignored. This can lead to confusing behavior where the "wrong" destination appears.

Fix: Use a single destination per type. If you need to route to different views based on String values, use a switch statement inside a single destination closure.

2. Forgetting Hashable conformance

All values pushed via NavigationLink(value:) or path.append() must conform to Hashable. If your model doesn't conform, you'll get a compile error. For structs with only basic stored properties, adding : Hashable is usually enough (the compiler synthesizes the conformance).

Fix: Add Hashable conformance. For classes or complex types, implement hash(into:) and == manually.

3. NavigationPath.codable returns nil

The .codable property returns nil if any value in the path doesn't conform to Codable. This is a runtime check -- you won't get a compiler warning.

Fix: Ensure all types pushed onto NavigationPath also conform to Codable if you need persistence. Consider using a typed array with a Codable enum instead.

4. Placing .navigationDestination outside NavigationStack

The .navigationDestination modifier must be placed on a view inside the NavigationStack, not on the NavigationStack itself or outside it. Placing it incorrectly means pushed values won't find a matching destination.

Fix: Always apply .navigationDestination to a view within the NavigationStack's content closure.

5. Nesting NavigationStacks

Placing a NavigationStack inside another NavigationStack creates two independent navigation hierarchies. You'll see double navigation bars and confusing behavior.

Fix: Use only one NavigationStack at the top of your view hierarchy. Child views should use NavigationLink and .navigationDestination without wrapping themselves in another stack.

6. Using NavigationLink with both "value" and "destination" styles in the same stack

Mixing NavigationLink("Title") { DestView() } (view-based) with NavigationLink(value: myValue) { Label() } (value-based) in the same NavigationStack is technically allowed but can cause confusion. View-based links don't participate in the path binding, so they won't update your @State path.

Fix: Pick one style per NavigationStack. For programmatic control, use value-based links exclusively.

7. Mutating path too rapidly

Appending multiple items to the path in rapid succession (e.g., in a loop without animation grouping) can sometimes cause visual glitches or unexpected behavior.

Fix: When setting multiple levels at once, prefer assigning the entire array: path = [.screen1, .screen2, .screen3] (for typed arrays) or use withAnimation to group changes.
