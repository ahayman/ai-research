The Composable Architecture (TCA) Navigation

A comprehensive guide to tree-based, stack-based, and presentation navigation patterns in SwiftUI using Point-Free's Composable Architecture.

Library Version

This guide covers swift-composable-architecture 1.x (the modern API with @Reducer macro, @ObservableState, and @Presents). The API has stabilized significantly since 1.0 and these patterns represent the recommended approach as of 2025.

Navigation as State

The core insight of TCA's navigation system is deceptively simple: navigation is just state. Every screen that can be shown, every sheet that can be presented, every alert that can appear — all are represented as values in your feature's state. When the state is nil, the destination is dismissed. When the state has a value, the destination is presented.

This has profound implications:

  • Deterministic behavior: Navigation is entirely driven by state mutations in your reducer. There are no side-channel imperative calls to present() or pushViewController().
  • Full testability: You can write unit tests that assert on every navigation transition by simply checking state changes.
  • Deep linking for free: To deep-link into any screen, you just construct the right state. No URL routing frameworks needed.
  • Automatic cleanup: When a destination is dismissed (state set to nil), TCA automatically cancels all in-flight effects for that feature.

TCA provides two paradigms for modeling navigation: tree-based and stack-based. Each has distinct strengths, and most real apps use both.

The key additions for navigation are:

  • @Presents — marks an optional child state property as a presentation destination (tree-based).
  • StackState<Path.State> — an ordered collection of screens for stack navigation.
  • .ifLet — a reducer operator that runs a child reducer when optional state is non-nil.
  • .forEach — a reducer operator that runs a path reducer for each element in a stack.

Tree-Based Navigation

Tree-based navigation models your app's screens as a tree of optional child states nested inside parent states. A parent feature owns a @Presents var destination: Destination.State? property. When that property is nil, no child is shown. When you assign a value, the child screen appears (as a pushed view, sheet, or cover).

Here is a complete example. A ContactListFeature can present an AddContactFeature as a sheet.

import ComposableArchitecture
import SwiftUI

// MARK: - Child Feature (AddContact)

@Reducer
struct AddContactFeature {
  @ObservableState
  struct State: Equatable {
    var name: String = ""
  }

  enum Action: BindableAction {
    case binding(BindingAction<State>)
    case saveButtonTapped
    case cancelButtonTapped
    case delegate(Delegate)

    enum Delegate {
      case didSave(name: String)
    }
  }

  var body: some ReducerOf<Self> {
    BindingReducer()
    Reduce { state, action in
      switch action {
      case .saveButtonTapped:
        return .send(.delegate(.didSave(name: state.name)))
      case .cancelButtonTapped, .binding, .delegate:
        return .none
      }
    }
  }
}

// MARK: - Parent Feature (ContactList)

@Reducer
struct ContactListFeature {
  @ObservableState
  struct State: Equatable {
    var contacts: IdentifiedArrayOf<String> = []
    // Tree-based: optional child state drives sheet presentation
    @Presents var addContact: AddContactFeature.State?
  }

  enum Action {
    case addButtonTapped
    // PresentationAction wraps child actions + dismiss
    case addContact(PresentationAction<AddContactFeature.Action>)
  }

  var body: some ReducerOf<Self> {
    Reduce { state, action in
      switch action {
      case .addButtonTapped:
        // Present the child by setting state to non-nil
        state.addContact = AddContactFeature.State()
        return .none

      case let .addContact(.presented(.delegate(.didSave(name)))):
        state.contacts.append(name)
        state.addContact = nil  // Dismiss the sheet
        return .none

      case .addContact(.presented(.cancelButtonTapped)):
        state.addContact = nil  // Dismiss the sheet
        return .none

      case .addContact:
        return .none
      }
    }
    // Connect child reducer: runs AddContactFeature when addContact != nil
    .ifLet(\.$addContact, action: \.addContact) {
      AddContactFeature()
    }
  }
}

// MARK: - Views

struct ContactListView: View {
  @Bindable var store: StoreOf<ContactListFeature>

  var body: some View {
    NavigationStack {
      List(store.contacts, id: \.self) { contact in
        Text(contact)
      }
      .navigationTitle("Contacts")
      .toolbar {
        Button("Add") { store.send(.addButtonTapped) }
      }
      // .sheet driven by the optional store
      .sheet(item: $store.scope(state: \.addContact, action: \.addContact)) { addStore in
        AddContactView(store: addStore)
      }
    }
  }
}

struct AddContactView: View {
  @Bindable var store: StoreOf<AddContactFeature>

  var body: some View {
    Form {
      TextField("Name", text: $store.name)
      Button("Save") { store.send(.saveButtonTapped) }
      Button("Cancel") { store.send(.cancelButtonTapped) }
    }
  }
}
Key Pattern

The parent's state holds @Presents var addContact: AddContactFeature.State?. Setting it to a value presents the sheet; setting it to nil dismisses it. The .ifLet(\.$addContact, action: \.addContact) line connects the child reducer so it runs only when the child is presented. TCA automatically cancels any in-flight effects from the child when the state becomes nil.

The @Presents Macro

@Presents is a property wrapper macro that marks an optional child state as a navigation destination. Under the hood, it creates a projected value (\.$destination) that .ifLet uses to observe and manage the child's lifecycle.

@ObservableState
struct State: Equatable {
  // Each @Presents property is an independent navigation destination
  @Presents var detail: DetailFeature.State?
  @Presents var alert: AlertState<Action.Alert>?
  @Presents var confirmDialog: ConfirmationDialogState<Action.ConfirmDialog>?
}

You can also use an enum Destination when a single UI element can present one of several screens (but only one at a time):

@Reducer
struct ParentFeature {
  @Reducer
  enum Destination {
    case detail(DetailFeature)
    case edit(EditFeature)
    case alert(AlertState<Action.Alert>)
  }

  @ObservableState
  struct State: Equatable {
    @Presents var destination: Destination.State?
  }

  enum Action {
    case destination(PresentationAction<Destination.Action>)
    case showDetailTapped
    case showEditTapped
  }

  var body: some ReducerOf<Self> {
    Reduce { state, action in
      switch action {
      case .showDetailTapped:
        state.destination = .detail(DetailFeature.State(id: "123"))
        return .none
      case .showEditTapped:
        state.destination = .edit(EditFeature.State())
        return .none
      case .destination:
        return .none
      }
    }
    .ifLet(\.$destination, action: \.destination)
  }
}
@Reducer enum

When you annotate an enum with @Reducer, TCA automatically synthesizes the State and Action enums and the body reducer for the destination. Each case maps to a child feature's reducer. This dramatically reduces boilerplate compared to earlier versions of TCA.

Sheet & Full-Screen Cover Presentation

TCA provides overloads of SwiftUI's .sheet and .fullScreenCover that accept a scoped store binding instead of a Binding<Bool> or Binding<Item?>.

@Reducer
struct HomeFeature {
  @Reducer
  enum Destination {
    case settings(SettingsFeature)
    case profile(ProfileFeature)
  }

  @ObservableState
  struct State: Equatable {
    @Presents var destination: Destination.State?
  }

  enum Action {
    case destination(PresentationAction<Destination.Action>)
    case settingsButtonTapped
    case profileButtonTapped
  }

  var body: some ReducerOf<Self> {
    Reduce { state, action in
      switch action {
      case .settingsButtonTapped:
        state.destination = .settings(SettingsFeature.State())
        return .none
      case .profileButtonTapped:
        state.destination = .profile(ProfileFeature.State())
        return .none
      case .destination:
        return .none
      }
    }
    .ifLet(\.$destination, action: \.destination)
  }
}

struct HomeView: View {
  @Bindable var store: StoreOf<HomeFeature>

  var body: some View {
    VStack(spacing: 20) {
      Button("Settings") { store.send(.settingsButtonTapped) }
      Button("Profile") { store.send(.profileButtonTapped) }
    }
    // Sheet for settings
    .sheet(
      item: $store.scope(state: \.destination?.settings, action: \.destination.settings)
    ) { settingsStore in
      SettingsView(store: settingsStore)
    }
    // Full-screen cover for profile
    .fullScreenCover(
      item: $store.scope(state: \.destination?.profile, action: \.destination.profile)
    ) { profileStore in
      ProfileView(store: profileStore)
    }
  }
}
SwiftUI Limitation

SwiftUI does not support presenting multiple sheets or covers simultaneously. If your destination enum has multiple cases that present as sheets, only one can be active at a time (which TCA's optional state naturally enforces). If you need truly independent presentations, use separate @Presents properties instead of a single enum.

Alerts & Confirmation Dialogs

TCA provides AlertState and ConfirmationDialogState types that describe alerts as pure data. Actions for each button are specified as enum cases, making alerts fully testable.

@Reducer
struct ItemFeature {
  @ObservableState
  struct State: Equatable {
    var item: Item
    @Presents var alert: AlertState<Action.Alert>?
    @Presents var confirmDialog: ConfirmationDialogState<Action.ConfirmDialog>?
  }

  enum Action {
    case deleteButtonTapped
    case archiveButtonTapped
    case alert(PresentationAction<Alert>)
    case confirmDialog(PresentationAction<ConfirmDialog>)

    enum Alert {
      case confirmDelete
    }

    enum ConfirmDialog {
      case confirmArchive
      case confirmDeleteInstead
    }
  }

  var body: some ReducerOf<Self> {
    Reduce { state, action in
      switch action {
      case .deleteButtonTapped:
        state.alert = AlertState {
          TextState("Delete Item?")
        } actions: {
          ButtonState(role: .destructive, action: .confirmDelete) {
            TextState("Delete")
          }
          ButtonState(role: .cancel) {
            TextState("Cancel")
          }
        } message: {
          TextState("This action cannot be undone.")
        }
        return .none

      case .archiveButtonTapped:
        state.confirmDialog = ConfirmationDialogState {
          TextState("Archive Options")
        } actions: {
          ButtonState(action: .confirmArchive) {
            TextState("Archive")
          }
          ButtonState(role: .destructive, action: .confirmDeleteInstead) {
            TextState("Delete Instead")
          }
          ButtonState(role: .cancel) {
            TextState("Cancel")
          }
        }
        return .none

      case .alert(.presented(.confirmDelete)):
        // Perform deletion logic
        return .none

      case .confirmDialog(.presented(.confirmArchive)):
        // Perform archive logic
        return .none

      case .confirmDialog(.presented(.confirmDeleteInstead)):
        // Perform deletion logic
        return .none

      case .alert, .confirmDialog:
        return .none
      }
    }
    .ifLet(\.$alert, action: \.alert)
    .ifLet(\.$confirmDialog, action: \.confirmDialog)
  }
}

struct ItemView: View {
  @Bindable var store: StoreOf<ItemFeature>

  var body: some View {
    VStack {
      Text(store.item.name)
      Button("Delete") { store.send(.deleteButtonTapped) }
      Button("Archive") { store.send(.archiveButtonTapped) }
    }
    .alert($store.scope(state: \.alert, action: \.alert))
    .confirmationDialog($store.scope(state: \.confirmDialog, action: \.confirmDialog))
  }
}

Stack-Based Navigation

Stack-based navigation models a flat collection of screens pushed onto a navigation stack. Instead of nesting child states inside parents, you maintain a StackState — essentially an array of heterogeneous screen states. This approach mirrors NavigationStack + NavigationPath in vanilla SwiftUI.

When to Choose Stack-Based

  • You have a list that drills into detail screens of varying types.
  • You need to support deep linking more than 2–3 levels deep.
  • The same type of screen can appear multiple times in the stack (e.g., folder → subfolder → subfolder).
  • You want to pop to root or pop to a specific screen programmatically.

StackState & StackAction

StackState is a special collection type from TCA that behaves like an array but assigns stable IDs to each element. StackAction is an enum that wraps child actions and includes a special .element(id:action:) case for routing actions to the correct child in the stack.

graph LR subgraph "StackState<Path.State>" E0["[0] .list(ListState)"] --> E1["[1] .detail(DetailState)"] E1 --> E2["[2] .edit(EditState)"] end subgraph "StackAction<Path.State, Path.Action>" A1[".element(id: 0, action: .list(...))"] A2[".element(id: 1, action: .detail(...))"] A3[".push(id:, state:)"] A4[".popFrom(id:)"] end style E0 fill:#e8f5e9,stroke:#34c759,color:#1d1d1f style E1 fill:#e8f5e9,stroke:#34c759,color:#1d1d1f style E2 fill:#e8f5e9,stroke:#34c759,color:#1d1d1f
Figure 4: StackState holds an ordered collection of heterogeneous screen states. StackAction routes actions to specific elements by ID.
@Reducer
struct RootFeature {
  // Define all possible screens in the stack
  @Reducer
  enum Path {
    case detail(DetailFeature)
    case edit(EditFeature)
    case settings(SettingsFeature)
  }

  @ObservableState
  struct State: Equatable {
    var items: IdentifiedArrayOf<Item> = []
    // The navigation stack state
    var path = StackState<Path.State>()
  }

  enum Action {
    case itemTapped(Item)
    case settingsButtonTapped
    // StackAction wraps child actions + push/pop
    case path(StackActionOf<Path>)
  }

  var body: some ReducerOf<Self> {
    Reduce { state, action in
      switch action {
      case let .itemTapped(item):
        // Push a detail screen onto the stack
        state.path.append(.detail(DetailFeature.State(item: item)))
        return .none

      case .settingsButtonTapped:
        state.path.append(.settings(SettingsFeature.State()))
        return .none

      // Handle delegate actions from children in the stack
      case let .path(.element(id: id, action: .detail(.delegate(.editTapped(item))))):
        state.path.append(.edit(EditFeature.State(item: item)))
        return .none

      case let .path(.element(id: _, action: .edit(.delegate(.didSave(item))))):
        // Update item in parent state and pop back
        state.items[id: item.id] = item
        // Pop the edit screen
        let _ = state.path.popLast()
        return .none

      case .path:
        return .none
      }
    }
    // Connect the Path reducer to the stack
    .forEach(\.path, action: \.path)
  }
}

NavigationStack in TCA

On the view side, you use SwiftUI's NavigationStack(path:root:destination:) with a store scoped to the path state and action. The destination closure switches on each case of Path.State to render the appropriate view.

struct RootView: View {
  @Bindable var store: StoreOf<RootFeature>

  var body: some View {
    NavigationStack(path: $store.scope(state: \.path, action: \.path)) {
      // Root view of the stack
      List(store.items) { item in
        Button(item.name) {
          store.send(.itemTapped(item))
        }
      }
      .navigationTitle("Items")
      .toolbar {
        Button("Settings") { store.send(.settingsButtonTapped) }
      }
    } destination: { pathStore in
      // Switch on the path case to render the correct view
      switch pathStore.case {
      case let .detail(detailStore):
        DetailView(store: detailStore)
      case let .edit(editStore):
        EditView(store: editStore)
      case let .settings(settingsStore):
        SettingsView(store: settingsStore)
      }
    }
  }
}
pathStore.case

The .case property is synthesized by the @Reducer macro on the Path enum. It returns a scoped store for whichever case is currently active, enabling exhaustive switching over all possible destinations in the stack.

Deep Linking

Because navigation is just state, deep linking is trivial: construct the desired StackState and you are done. No URL routing, no coordinator chains — just data.

// Deep link to: Root → Detail → Edit
let deepLinkedState = RootFeature.State(
  items: myItems,
  path: StackState([
    .detail(DetailFeature.State(item: targetItem)),
    .edit(EditFeature.State(item: targetItem)),
  ])
)

// Create a store with that state and the app opens right there
let store = Store(initialState: deepLinkedState) {
  RootFeature()
}

// Or, at runtime, mutate path to deep link:
store.send(.deepLink(to: targetItem)) // in your reducer:
// case let .deepLink(to: item):
//   state.path.removeAll()
//   state.path.append(.detail(DetailFeature.State(item: item)))
//   state.path.append(.edit(EditFeature.State(item: item)))
//   return .none
Deep Linking Advantage

This is one of TCA's strongest selling points for navigation. In vanilla SwiftUI, deep linking into a NavigationStack multiple levels deep is notoriously difficult, often requiring delays and workarounds. In TCA, you just construct the state array and everything renders correctly.

Enum-Based Paths with @Reducer

The @Reducer enum pattern is the modern, idiomatic way to define all possible navigation destinations in a stack. Each enum case holds a child feature's reducer, and TCA synthesizes the State and Action enums automatically.

// The @Reducer macro on an enum auto-generates:
// - Path.State enum with associated values for each case
// - Path.Action enum with associated values for each case
// - Path.body reducer that routes to the correct child reducer

@Reducer
enum Path {
  case featureA(FeatureA)
  case featureB(FeatureB)
  case featureC(FeatureC)
}

TCACoordinators Library

TCACoordinators is a community library by John Patrick Morgan that brings the coordinator pattern to TCA. It builds on top of TCA's native navigation tools and adds a higher-level abstraction for managing complex flows.

Why Use It?

  • Screen isolation: Individual screen features have zero knowledge of the navigation flow. They just send delegate actions.
  • Reusable coordinators: Break your app's navigation into composable coordinator units.
  • Complex flows: Onboarding wizards, authentication flows, multi-step forms become clean to manage.
  • SwiftUI workarounds: Handles edge cases where SwiftUI doesn't support presenting multiple screens in a single state update (important for deep linking).
import TCACoordinators

// Define screens as an enum
@Reducer
enum Screen {
  case login(LoginFeature)
  case home(HomeFeature)
  case profile(ProfileFeature)
}

// Coordinator manages the flow
@Reducer
struct AppCoordinator {
  @ObservableState
  struct State: Equatable, IndexedRouterState {
    var routes: [Route<Screen.State>] = [
      .root(.login(LoginFeature.State()), embedInNavigationView: true)
    ]
  }

  enum Action: IndexedRouterAction {
    case routeAction(_ index: Int, action: Screen.Action)
    case updateRoutes([Route<Screen.State>])
  }

  var body: some ReducerOf<Self> {
    Reduce { state, action in
      switch action {
      case .routeAction(_, action: .login(.delegate(.didLogin))):
        state.routes = [
          .root(.home(HomeFeature.State()), embedInNavigationView: true)
        ]
        return .none

      case .routeAction(_, action: .home(.delegate(.showProfile))):
        state.routes.push(.profile(ProfileFeature.State()))
        return .none

      default:
        return .none
      }
    }
    .forEachRoute(\.routes, action: \.routeAction)
  }
}

// View uses TCARouter
struct AppCoordinatorView: View {
  let store: StoreOf<AppCoordinator>

  var body: some View {
    TCARouter(store.scope(state: \.routes, action: \.routeAction)) { screen in
      switch screen.case {
      case let .login(store): LoginView(store: store)
      case let .home(store): HomeView(store: store)
      case let .profile(store): ProfileView(store: store)
      }
    }
  }
}
TCACoordinators vs. Native TCA Navigation

TCACoordinators is a complementary layer on top of TCA's native tools. For simple apps, native StackState or @Presents is sufficient. Consider TCACoordinators when you have complex multi-step flows, need to swap entire navigation hierarchies (e.g., login → main app), or want maximum screen isolation.

Testing Navigation

This is where TCA's navigation approach truly shines. Because navigation is state, you can write exhaustive unit tests that assert on every screen transition, without launching the simulator or touching any UI.

Testing Tree-Based Navigation

import ComposableArchitecture
import XCTest

@MainActor
final class ContactListTests: XCTestCase {
  func testAddContactFlow() async {
    let store = TestStore(initialState: ContactListFeature.State()) {
      ContactListFeature()
    }

    // 1. Tap "Add" — assert sheet is presented
    await store.send(.addButtonTapped) {
      $0.addContact = AddContactFeature.State()
    }

    // 2. Type a name in the child
    await store.send(\.addContact.binding.name, "Alice") {
      $0.addContact?.name = "Alice"
    }

    // 3. Tap "Save" — assert contact added and sheet dismissed
    await store.send(\.addContact.saveButtonTapped)
    await store.receive(\.addContact.delegate.didSave) {
      $0.contacts = ["Alice"]
      $0.addContact = nil  // Sheet dismissed!
    }
  }

  func testCancelDismissesSheet() async {
    let store = TestStore(
      initialState: ContactListFeature.State(
        addContact: AddContactFeature.State(name: "Draft")
      )
    ) {
      ContactListFeature()
    }

    // Cancel dismisses without saving
    await store.send(\.addContact.cancelButtonTapped) {
      $0.addContact = nil
    }
  }
}

Why This Matters

These tests run in milliseconds with no simulator. You are testing the actual navigation logic — which screens appear, which data they receive, what happens when the user taps buttons. This is not possible with vanilla SwiftUI navigation without UI tests, which are orders of magnitude slower and more brittle.

Complete Mini-App: Task Manager

Here is a complete, self-contained TCA application demonstrating tree-based and stack-based navigation working together. This code compiles with Xcode and the swift-composable-architecture package dependency.

// TaskManagerApp.swift
// Requires: swift-composable-architecture ~> 1.0
// Add via SPM: https://github.com/pointfreeco/swift-composable-architecture

import ComposableArchitecture
import SwiftUI

// MARK: - Model

struct TaskItem: Equatable, Identifiable {
  let id: UUID
  var title: String
  var isComplete: Bool = false
  var notes: String = ""
}

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// MARK: - Add Task Feature (tree-based, presented as sheet)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

@Reducer
struct AddTaskFeature {
  @ObservableState
  struct State: Equatable {
    var title: String = ""
    var notes: String = ""
  }

  enum Action: BindableAction {
    case binding(BindingAction<State>)
    case saveButtonTapped
    case cancelButtonTapped
    case delegate(Delegate)

    enum Delegate {
      case didSave(TaskItem)
    }
  }

  @Dependency(\.uuid) var uuid

  var body: some ReducerOf<Self> {
    BindingReducer()
    Reduce { state, action in
      switch action {
      case .saveButtonTapped:
        let task = TaskItem(
          id: uuid(),
          title: state.title,
          notes: state.notes
        )
        return .send(.delegate(.didSave(task)))
      case .cancelButtonTapped, .binding, .delegate:
        return .none
      }
    }
  }
}

struct AddTaskView: View {
  @Bindable var store: StoreOf<AddTaskFeature>

  var body: some View {
    NavigationStack {
      Form {
        TextField("Title", text: $store.title)
        TextField("Notes", text: $store.notes, axis: .vertical)
          .lineLimit(3...6)
      }
      .navigationTitle("New Task")
      .toolbar {
        ToolbarItem(placement: .cancellationAction) {
          Button("Cancel") { store.send(.cancelButtonTapped) }
        }
        ToolbarItem(placement: .confirmationAction) {
          Button("Save") { store.send(.saveButtonTapped) }
            .disabled(store.title.isEmpty)
        }
      }
    }
  }
}

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// MARK: - Task Detail Feature (stack destination)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

@Reducer
struct TaskDetailFeature {
  @ObservableState
  struct State: Equatable {
    var task: TaskItem
    @Presents var alert: AlertState<Action.Alert>?
  }

  enum Action: BindableAction {
    case binding(BindingAction<State>)
    case toggleComplete
    case deleteButtonTapped
    case editButtonTapped
    case alert(PresentationAction<Alert>)
    case delegate(Delegate)

    enum Alert {
      case confirmDelete
    }
    enum Delegate {
      case didDelete(TaskItem)
      case editTask(TaskItem)
      case didUpdate(TaskItem)
    }
  }

  var body: some ReducerOf<Self> {
    BindingReducer()
    Reduce { state, action in
      switch action {
      case .toggleComplete:
        state.task.isComplete.toggle()
        return .send(.delegate(.didUpdate(state.task)))

      case .deleteButtonTapped:
        state.alert = AlertState {
          TextState("Delete \"\(state.task.title)\"?")
        } actions: {
          ButtonState(role: .destructive, action: .confirmDelete) {
            TextState("Delete")
          }
          ButtonState(role: .cancel) { TextState("Cancel") }
        }
        return .none

      case .alert(.presented(.confirmDelete)):
        return .send(.delegate(.didDelete(state.task)))

      case .editButtonTapped:
        return .send(.delegate(.editTask(state.task)))

      case .binding, .alert, .delegate:
        return .none
      }
    }
    .ifLet(\.$alert, action: \.alert)
  }
}

struct TaskDetailView: View {
  @Bindable var store: StoreOf<TaskDetailFeature>

  var body: some View {
    Form {
      Section("Task") {
        Text(store.task.title).font(.headline)
        if !store.task.notes.isEmpty {
          Text(store.task.notes).foregroundStyle(.secondary)
        }
      }
      Section {
        Button {
          store.send(.toggleComplete)
        } label: {
          Label(
            store.task.isComplete ? "Completed" : "Mark Complete",
            systemImage: store.task.isComplete ? "checkmark.circle.fill" : "circle"
          )
        }
      }
      Section {
        Button("Edit Task") { store.send(.editButtonTapped) }
        Button("Delete Task", role: .destructive) { store.send(.deleteButtonTapped) }
      }
    }
    .navigationTitle("Details")
    .alert($store.scope(state: \.alert, action: \.alert))
  }
}

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// MARK: - Edit Task Feature (stack destination)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

@Reducer
struct EditTaskFeature {
  @ObservableState
  struct State: Equatable {
    var task: TaskItem
  }

  enum Action: BindableAction {
    case binding(BindingAction<State>)
    case saveButtonTapped
    case delegate(Delegate)

    enum Delegate {
      case didSave(TaskItem)
    }
  }

  var body: some ReducerOf<Self> {
    BindingReducer()
    Reduce { state, action in
      switch action {
      case .saveButtonTapped:
        return .send(.delegate(.didSave(state.task)))
      case .binding, .delegate:
        return .none
      }
    }
  }
}

struct EditTaskView: View {
  @Bindable var store: StoreOf<EditTaskFeature>

  var body: some View {
    Form {
      TextField("Title", text: $store.task.title)
      TextField("Notes", text: $store.task.notes, axis: .vertical)
        .lineLimit(3...6)
      Toggle("Completed", isOn: $store.task.isComplete)
    }
    .navigationTitle("Edit")
    .toolbar {
      Button("Save") { store.send(.saveButtonTapped) }
    }
  }
}

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// MARK: - App Root Feature (combines tree + stack navigation)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

@Reducer
struct AppFeature {
  // Stack destinations (enum-based path)
  @Reducer
  enum Path {
    case detail(TaskDetailFeature)
    case edit(EditTaskFeature)
  }

  @ObservableState
  struct State: Equatable {
    var tasks: IdentifiedArrayOf<TaskItem> = []
    var path = StackState<Path.State>()
    // Tree-based: sheet presentation
    @Presents var addTask: AddTaskFeature.State?
  }

  enum Action {
    case addButtonTapped
    case taskTapped(TaskItem)
    // Tree-based action
    case addTask(PresentationAction<AddTaskFeature.Action>)
    // Stack-based action
    case path(StackActionOf<Path>)
  }

  var body: some ReducerOf<Self> {
    Reduce { state, action in
      switch action {

      // --- Tree-based: Add Task Sheet ---
      case .addButtonTapped:
        state.addTask = AddTaskFeature.State()
        return .none

      case let .addTask(.presented(.delegate(.didSave(task)))):
        state.tasks.append(task)
        state.addTask = nil
        return .none

      case .addTask(.presented(.cancelButtonTapped)):
        state.addTask = nil
        return .none

      case .addTask:
        return .none

      // --- Stack-based: Push Detail ---
      case let .taskTapped(task):
        state.path.append(.detail(TaskDetailFeature.State(task: task)))
        return .none

      // Handle delegate actions from stack children
      case let .path(.element(id: _, action: .detail(.delegate(.didUpdate(task))))):
        state.tasks[id: task.id] = task
        return .none

      case let .path(.element(id: _, action: .detail(.delegate(.didDelete(task))))):
        state.tasks.remove(id: task.id)
        let _ = state.path.popLast()
        return .none

      case let .path(.element(id: _, action: .detail(.delegate(.editTask(task))))):
        state.path.append(.edit(EditTaskFeature.State(task: task)))
        return .none

      case let .path(.element(id: _, action: .edit(.delegate(.didSave(task))))):
        state.tasks[id: task.id] = task
        // Also update the detail screen below in the stack
        if let detailIdx = state.path.ids.dropLast().last {
          state.path[id: detailIdx, case: \Path.State.detail]?.task = task
        }
        let _ = state.path.popLast()
        return .none

      case .path:
        return .none
      }
    }
    // Tree-based child reducer
    .ifLet(\.$addTask, action: \.addTask) {
      AddTaskFeature()
    }
    // Stack-based path reducer
    .forEach(\.path, action: \.path)
  }
}

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// MARK: - App Root View
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

struct AppView: View {
  @Bindable var store: StoreOf<AppFeature>

  var body: some View {
    NavigationStack(path: $store.scope(state: \.path, action: \.path)) {
      List(store.tasks) { task in
        Button {
          store.send(.taskTapped(task))
        } label: {
          HStack {
            Image(systemName: task.isComplete ? "checkmark.circle.fill" : "circle")
              .foregroundStyle(task.isComplete ? .green : .secondary)
            VStack(alignment: .leading) {
              Text(task.title)
                .strikethrough(task.isComplete)
              if !task.notes.isEmpty {
                Text(task.notes)
                  .font(.caption)
                  .foregroundStyle(.secondary)
                  .lineLimit(1)
              }
            }
          }
        }
      }
      .navigationTitle("Tasks")
      .toolbar {
        Button("Add", systemImage: "plus") {
          store.send(.addButtonTapped)
        }
      }
      // Tree-based: sheet
      .sheet(item: $store.scope(state: \.addTask, action: \.addTask)) { addStore in
        AddTaskView(store: addStore)
      }
    } destination: { pathStore in
      // Stack-based: switch on path cases
      switch pathStore.case {
      case let .detail(store):
        TaskDetailView(store: store)
      case let .edit(store):
        EditTaskView(store: store)
      }
    }
  }
}

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// MARK: - App Entry Point
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

@main
struct TaskManagerApp: App {
  var body: some Scene {
    WindowGroup {
      AppView(
        store: Store(initialState: AppFeature.State()) {
          AppFeature()
        }
      )
    }
  }
}
Architecture Summary

This mini-app uses tree-based navigation for the "Add Task" sheet (presented modally, dismissed by setting state to nil) and stack-based navigation for the detail/edit drill-down (pushed onto the NavigationStack via StackState). The two approaches compose naturally within a single feature.

Tradeoff Analysis

DimensionTCA NavigationVanilla SwiftUI
Learning Curve Steep. Requires understanding TCA's reducer composition, macros (@Reducer, @Presents, @ObservableState), and navigation operators (.ifLet, .forEach). Moderate. SwiftUI's own navigation APIs have their own complexity but are more widely documented.
Testability Excellent. Every navigation transition is a state mutation you can assert on in unit tests. TestStore makes this ergonomic. Poor for navigation logic. You must rely on UI tests or manually instrument navigation state.
Deep Linking Trivial. Construct the state, and the UI renders. Works reliably at any depth. Difficult. NavigationPath deep linking beyond 2-3 levels is buggy; requires workarounds.
Correctness High. Exhaustive action handling prevents forgotten cases. Automatic effect cancellation on dismissal prevents leaks. Moderate. Easy to forget to cancel tasks or handle edge cases.
Boilerplate Moderate-high. Each feature needs State, Action, body reducer. The @Reducer macro reduces it significantly compared to pre-1.0 TCA. Low-moderate. SwiftUI views can be self-contained with @State and @Observable.
Debugging Good. _printChanges() logs every action and state change. Deterministic behavior makes issues reproducible. Mixed. SwiftUI's navigation state can be opaque and hard to inspect.
Performance Good. @ObservableState provides minimal re-renders. Stack elements are lazily initialized. Good. Native SwiftUI optimization.
Team Adoption Requires the entire team to understand TCA. Not something you can adopt partially for navigation alone. Everyone already knows it.
Honest Assessment of the Learning Curve

TCA navigation has a significant learning curve. You need to understand reducers, actions, state composition, case key paths, the .ifLet and .forEach operators, presentation actions, delegate actions, store scoping, and the various macros. For a team new to TCA, expect 2-4 weeks before developers are comfortable writing navigation code independently. The payoff is real — testability, correctness, and deep linking — but the investment is substantial.

When to Use TCA Navigation

Use TCA Navigation When...

  • Your team is already using or committed to TCA
  • Navigation logic is complex (multi-step flows, conditional routing)
  • Deep linking is a core requirement
  • You need exhaustive unit tests for navigation flows
  • Your app has shared state across navigation destinations
  • You want deterministic, reproducible navigation behavior
  • The app is medium-to-large scale with multiple developers

Stick with Vanilla SwiftUI When...

  • Your app is small or a prototype
  • Navigation is simple (a few screens, straightforward flows)
  • Your team doesn't know TCA and timeline is tight
  • You don't need to unit-test navigation transitions
  • Deep linking is not a requirement
  • You prefer Apple-native patterns with minimal dependencies
  • You are building a widget, extension, or embedded SwiftUI view
