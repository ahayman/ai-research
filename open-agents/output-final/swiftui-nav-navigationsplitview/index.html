NavigationSplitView

A container view that presents two or three columns of content side by side, where selections in leading columns drive what appears in trailing columns. The multi-column navigation pattern for iPadOS, macOS, and adaptive iOS layouts.

iOS 16+ / macOS 13+ / tvOS 16+ | Multi-column navigation | Replaces UISplitViewController

What is NavigationSplitView?

NavigationSplitView is SwiftUI's multi-column navigation container, introduced in iOS 16 (WWDC 2022). It replaces the older pattern of wrapping a NavigationView with .navigationViewStyle(.columns) and provides a first-class API for building sidebar-driven layouts that are native to iPadOS and macOS.

At its core, NavigationSplitView presents either two columns (sidebar + detail) or three columns (sidebar + content + detail). Selections made in the leading columns determine what appears in the trailing columns. On compact-width devices like iPhones, the split view automatically collapses into a stack-based navigation flow.

This is the SwiftUI equivalent of UISplitViewController from UIKit, but with a declarative, state-driven API that integrates naturally with SwiftUI's data flow.

Key Mental Model

Think of NavigationSplitView as 2D panel-based navigation: columns sit side by side, and selecting an item in one column updates the adjacent column. This is fundamentally different from NavigationStack's 3D depth-based navigation, where views push on top of each other.

NavigationSplitView vs NavigationStack

Understanding when to use each navigation container is critical. They solve different problems and serve different form factors.

Two-Column vs Three-Column Layout

NavigationSplitView supports two initializers, one for each column configuration:

Two-Column: Sidebar + Detail

The simpler form. A sidebar on the leading edge and a detail view on the trailing edge. Best for apps where selecting an item in the sidebar immediately shows full detail. Think: Settings app, simple mail client, or a notes app.

Three-Column: Sidebar + Content + Detail

Adds a middle "content" column between sidebar and detail. This is ideal when you have a two-level hierarchy: the sidebar selects a category, the content column lists items in that category, and the detail column shows the selected item. Think: Mail.app (mailboxes → message list → message detail) or Files.app.

Basic Two-Column NavigationSplitView

iOS 16+ iPadOS 16+ macOS 13+

The simplest NavigationSplitView takes a sidebar closure and a detail closure. This creates a two-column layout where you typically place a List in the sidebar and show the selected item's detail in the trailing column.

Swift 
import SwiftUI

struct Fruit: Identifiable, Hashable {
    let id = UUID()
    let name: String
    let emoji: String
    let color: Color
}

struct BasicTwoColumnView: View {
    let fruits = [
        Fruit(name: "Apple", emoji: "🍎", color: .red),
        Fruit(name: "Banana", emoji: "🍌", color: .yellow),
        Fruit(name: "Blueberry", emoji: "🫐", color: .blue),
        Fruit(name: "Mango", emoji: "🥭", color: .orange),
    ]

    @State private var selectedFruit: Fruit?

    var body: some View {
        NavigationSplitView {
            List(fruits, selection: $selectedFruit) { fruit in
                NavigationLink(value: fruit) {
                    Label(fruit.name, systemImage: "leaf")
                }
            }
            .navigationTitle("Fruits")
        } detail: {
            if let fruit = selectedFruit {
                VStack(spacing: 20) {
                    Text(fruit.emoji)
                        .font(.system(size: 80))
                    Text(fruit.name)
                        .font(.largeTitle.bold())
                    RoundedRectangle(cornerRadius: 12)
                        .fill(fruit.color.gradient)
                        .frame(width: 200, height: 200)
                }
                .navigationTitle(fruit.name)
            } else {
                ContentUnavailableView(
                    "Select a Fruit",
                    systemImage: "leaf.circle",
                    description: Text("Pick a fruit from the sidebar.")
                )
            }
        }
    }
}
How it works

The selection parameter on List binds to $selectedFruit. When the user taps a row, SwiftUI updates selectedFruit, and the detail closure re-evaluates to show the chosen fruit. On iPad/Mac, both columns are visible simultaneously. On iPhone, tapping a row pushes the detail view onto a navigation stack.

Three-Column Layout

iOS 16+ iPadOS 16+ macOS 13+

The three-column initializer adds a content closure between sidebar and detail. Two separate selection bindings drive the hierarchy: one for the sidebar-to-content relationship, and one for the content-to-detail relationship.

Swift 
import SwiftUI

struct Category: Identifiable, Hashable {
    let id = UUID()
    let name: String
    let symbol: String
    let items: [String]
}

struct ThreeColumnView: View {
    let categories = [
        Category(name: "Fruits", symbol: "leaf",
                 items: ["Apple", "Banana", "Cherry"]),
        Category(name: "Vegetables", symbol: "carrot",
                 items: ["Carrot", "Broccoli", "Spinach"]),
        Category(name: "Grains", symbol: "leaf.circle",
                 items: ["Rice", "Wheat", "Oats"]),
    ]

    @State private var selectedCategory: Category?
    @State private var selectedItem: String?

    var body: some View {
        NavigationSplitView {
            // SIDEBAR: category list
            List(categories, selection: $selectedCategory) { category in
                NavigationLink(value: category) {
                    Label(category.name, systemImage: category.symbol)
                }
            }
            .navigationTitle("Food Groups")
        } content: {
            // CONTENT: items in selected category
            if let category = selectedCategory {
                List(category.items, id: \.self, selection: $selectedItem) { item in
                    NavigationLink(value: item) {
                        Text(item)
                    }
                }
                .navigationTitle(category.name)
            } else {
                ContentUnavailableView(
                    "Select a Category",
                    systemImage: "sidebar.left"
                )
            }
        } detail: {
            // DETAIL: selected item info
            if let item = selectedItem {
                VStack {
                    Text(item)
                        .font(.largeTitle.bold())
                    Text("Details about \(item) would go here.")
                        .foregroundStyle(.secondary)
                }
                .navigationTitle(item)
            } else {
                ContentUnavailableView(
                    "Select an Item",
                    systemImage: "doc.text",
                    description: Text("Choose something from the list.")
                )
            }
        }
    }
}

Notice the two separate @State bindings: selectedCategory drives which items appear in the content column, and selectedItem drives the detail. When the user changes the category, you will typically want to reset selectedItem to nil using an .onChange(of:) modifier.

Selection-Driven Navigation with Bindings

The binding flow between columns is the core mechanism of NavigationSplitView. Understanding how selection state propagates is essential for building correct multi-column navigation.

Figure 3: Selection binding flow between columns. Changes cascade left-to-right through @State.
Swift 
import SwiftUI

struct Email: Identifiable, Hashable {
    let id = UUID()
    let subject: String
    let sender: String
    let body: String
    let isRead: Bool
}

enum Mailbox: String, CaseIterable, Identifiable {
    case inbox, sent, drafts, trash
    var id: String { rawValue }
    var symbol: String {
        switch self {
        case .inbox: "tray"
        case .sent: "paperplane"
        case .drafts: "doc"
        case .trash: "trash"
        }
    }
}

struct MailAppView: View {
    @State private var selectedMailbox: Mailbox? = .inbox
    @State private var selectedEmail: Email?

    // Sample data
    let emails: [Mailbox: [Email]] = [
        .inbox: [
            Email(subject: "Meeting Tomorrow", sender: "Alice",
                  body: "Don't forget about our meeting.", isRead: false),
            Email(subject: "Project Update", sender: "Bob",
                  body: "Here's the latest on the project.", isRead: true),
        ],
        .sent: [
            Email(subject: "Re: Hello", sender: "Me",
                  body: "Thanks for reaching out!", isRead: true),
        ],
        .drafts: [],
        .trash: [],
    ]

    var body: some View {
        NavigationSplitView {
            List(Mailbox.allCases, selection: $selectedMailbox) { mailbox in
                NavigationLink(value: mailbox) {
                    Label(mailbox.rawValue.capitalized,
                          systemImage: mailbox.symbol)
                }
            }
            .navigationTitle("Mail")
        } content: {
            if let mailbox = selectedMailbox,
               let mailboxEmails = emails[mailbox] {
                if mailboxEmails.isEmpty {
                    ContentUnavailableView(
                        "No Messages",
                        systemImage: "tray"
                    )
                } else {
                    List(mailboxEmails, selection: $selectedEmail) { email in
                        NavigationLink(value: email) {
                            VStack(alignment: .leading) {
                                Text(email.subject)
                                    .fontWeight(email.isRead ? .regular : .bold)
                                Text(email.sender)
                                    .font(.caption)
                                    .foregroundStyle(.secondary)
                            }
                        }
                    }
                    .navigationTitle(mailbox.rawValue.capitalized)
                }
            } else {
                ContentUnavailableView(
                    "Select a Mailbox",
                    systemImage: "sidebar.left"
                )
            }
        } detail: {
            if let email = selectedEmail {
                VStack(alignment: .leading, spacing: 16) {
                    Text(email.subject)
                        .font(.title.bold())
                    Text("From: \(email.sender)")
                        .foregroundStyle(.secondary)
                    Divider()
                    Text(email.body)
                    Spacer()
                }
                .padding()
                .navigationTitle(email.subject)
            } else {
                ContentUnavailableView(
                    "Select a Message",
                    systemImage: "envelope"
                )
            }
        }
        .onChange(of: selectedMailbox) {
            selectedEmail = nil  // Reset detail when mailbox changes
        }
    }
}
Important: Reset child selections

When the parent selection changes (e.g., switching mailboxes), always reset the child selection to nil using .onChange(of:). Otherwise, the detail column may show stale data from the previous category. This is shown in the .onChange(of: selectedMailbox) modifier above.

Combining NavigationSplitView with NavigationStack

NavigationSplitView limits you to at most three levels of columns. If you need deeper navigation within the detail column (e.g., tapping an item to see sub-items, then sub-sub-items), embed a NavigationStack inside the detail closure.

Note

NavigationSplitView automatically wraps each column's root view inside a NavigationStack internally. You only need to add an explicit NavigationStack when you want to control a NavigationPath or push additional views beyond what the columns provide.

Swift 
import SwiftUI

struct Recipe: Identifiable, Hashable {
    let id = UUID()
    let name: String
    let ingredients: [String]
}

struct SplitWithStackView: View {
    let recipes = [
        Recipe(name: "Pasta", ingredients: ["Noodles", "Tomato Sauce", "Basil"]),
        Recipe(name: "Salad", ingredients: ["Lettuce", "Croutons", "Dressing"]),
    ]

    @State private var selectedRecipe: Recipe?
    @State private var detailPath = NavigationPath()

    var body: some View {
        NavigationSplitView {
            List(recipes, selection: $selectedRecipe) { recipe in
                NavigationLink(value: recipe) {
                    Text(recipe.name)
                }
            }
            .navigationTitle("Recipes")
        } detail: {
            // Embed NavigationStack for deep drill-down in detail
            NavigationStack(path: $detailPath) {
                if let recipe = selectedRecipe {
                    List {
                        Section("Ingredients") {
                            ForEach(recipe.ingredients, id: \.self) { ingredient in
                                NavigationLink(value: ingredient) {
                                    Text(ingredient)
                                }
                            }
                        }
                    }
                    .navigationTitle(recipe.name)
                    .navigationDestination(for: String.self) { ingredient in
                        Text("Details about \(ingredient)")
                            .font(.title)
                            .navigationTitle(ingredient)
                    }
                } else {
                    ContentUnavailableView(
                        "Select a Recipe",
                        systemImage: "fork.knife"
                    )
                }
            }
        }
        .onChange(of: selectedRecipe) {
            detailPath.removeLast(detailPath.count)  // Reset stack when selection changes
        }
    }
}

The NavigationStack(path:) in the detail column gives you full control over a back-stack within the detail area. When the sidebar selection changes, we clear the detail stack to avoid showing stale pushed views from a different recipe.

Column Width Customization

iOS 16+

By default, SwiftUI uses system-standard widths for each column. You can customize these with the .navigationSplitViewColumnWidth() modifier, applied to the root view inside each column closure.

Swift 
struct CustomColumnWidthView: View {
    @State private var selectedItem: String?

    var body: some View {
        NavigationSplitView {
            List(["Item A", "Item B", "Item C"], id: \.self,
                 selection: $selectedItem) { item in
                Text(item)
            }
            .navigationTitle("Sidebar")
            // Fixed width for sidebar
            .navigationSplitViewColumnWidth(200)
        } detail: {
            Text(selectedItem ?? "Select an item")
                .navigationTitle("Detail")
        }
    }
}

// Flexible width with min/ideal/max:
struct FlexibleWidthView: View {
    @State private var selectedItem: String?

    var body: some View {
        NavigationSplitView {
            List(["One", "Two", "Three"], id: \.self,
                 selection: $selectedItem) { item in
                Text(item)
            }
            .navigationTitle("Sidebar")
            // Flexible: user can resize between min and max
            .navigationSplitViewColumnWidth(
                min: 180, ideal: 220, max: 320
            )
        } content: {
            Text("Content column")
                .navigationSplitViewColumnWidth(
                    min: 200, ideal: 300, max: 400
                )
        } detail: {
            Text(selectedItem ?? "Detail")
        }
    }
}
Platform Caveat

The .navigationSplitViewColumnWidth() modifier is a suggestion to the system. SwiftUI may ignore it in certain contexts. In particular, sidebar column width customization has had inconsistent behavior on iPadOS 16 and 17 -- it may not be honored for the sidebar column specifically. The content and detail columns are more reliable.

Programmatic Navigation

Because NavigationSplitView is selection-driven, programmatic navigation means writing to the selection state properties. This is simpler than NavigationStack's path-based approach for column-level navigation, but you can combine both for full control.

Swift 
import SwiftUI

struct ProgrammaticNavView: View {
    let items = ["Alpha", "Beta", "Gamma", "Delta"]

    @State private var selectedItem: String?
    @State private var columnVisibility: NavigationSplitViewVisibility = .all

    var body: some View {
        NavigationSplitView(columnVisibility: $columnVisibility) {
            List(items, id: \.self, selection: $selectedItem) { item in
                Text(item)
            }
            .navigationTitle("Items")
        } detail: {
            VStack(spacing: 20) {
                if let item = selectedItem {
                    Text("Selected: \(item)")
                        .font(.title)
                } else {
                    Text("Nothing selected")
                        .foregroundStyle(.secondary)
                }

                // Programmatically select an item
                Button("Select Gamma") {
                    selectedItem = "Gamma"
                }

                // Programmatically hide sidebar
                Button("Show Detail Only") {
                    columnVisibility = .detailOnly
                }

                // Show all columns
                Button("Show All Columns") {
                    columnVisibility = .all
                }
            }
            .navigationTitle("Detail")
        }
    }
}

Setting selectedItem = "Gamma" programmatically is equivalent to the user tapping "Gamma" in the sidebar list. Similarly, writing to columnVisibility hides or shows columns programmatically -- useful for onboarding flows or deep-link scenarios.

Swift 
import SwiftUI

struct AdaptiveNavView: View {
    @Environment(\.horizontalSizeClass) private var sizeClass
    @State private var selectedItem: String?
    @State private var preferredColumn: NavigationSplitViewColumn = .sidebar

    let items = ["Dashboard", "Profile", "Settings", "Help"]

    var body: some View {
        // preferredCompactColumn controls which column appears
        // when the split view collapses on compact devices (iOS 17+)
        NavigationSplitView(
            preferredCompactColumn: $preferredColumn
        ) {
            List(items, id: \.self, selection: $selectedItem) { item in
                Text(item)
            }
            .navigationTitle("Menu")
        } detail: {
            if let item = selectedItem {
                VStack {
                    Text(item)
                        .font(.largeTitle)

                    if sizeClass == .compact {
                        Text("Running in compact mode")
                            .font(.caption)
                            .foregroundStyle(.secondary)
                    }
                }
                .navigationTitle(item)
            } else {
                ContentUnavailableView(
                    "Welcome",
                    systemImage: "hand.wave",
                    description: Text("Select an item from the sidebar.")
                )
            }
        }
    }
}
preferredCompactColumn (iOS 17+)

By default, when a NavigationSplitView collapses on compact devices, SwiftUI shows the sidebar. Starting with iOS 17, you can use the preferredCompactColumn parameter to control which column appears initially. For example, setting it to .detail shows the detail view first on iPhone, which is useful for apps where the detail content is more important than the sidebar list.

NavigationSplitView Styles

The .navigationSplitViewStyle() modifier controls how the split view allocates space between columns. There are three built-in styles:

Style Behavior Best For

struct StyledSplitView: View {
    @State private var selectedItem: String?

    var body: some View {
        NavigationSplitView {
            List(["Photos", "Videos", "Albums"], id: \.self,
                 selection: $selectedItem) { item in
                Label(item, systemImage: "photo")
            }
            .navigationTitle("Library")
        } detail: {
            Text(selectedItem ?? "Select something")
        }
        // .balanced gives equal space to sidebar and detail
        .navigationSplitViewStyle(.balanced)
    }
}
Visual Difference

With .prominentDetail (the default on many platforms), the sidebar slides over the detail content like a sheet. With .balanced, the sidebar and detail share the available space side by side, and the detail column shrinks to accommodate the sidebar. For apps like Mail where you want both columns visible at once, .balanced is the right choice.

Compact Size Class Behavior

When a NavigationSplitView runs in a compact horizontal size class, it automatically collapses into a single-column stack-based navigation. This includes:

  • All iPhones in portrait orientation
  • iPhone Max models in portrait (landscape gets regular size class)
  • iPad in Slide Over mode
  • iPad in 1/3 split screen
  • Any multitasking scenario where the app window is narrow

In compact mode, the columns become screens in a navigation stack. The sidebar becomes the root, the content column (if present) is pushed when a sidebar item is selected, and the detail is pushed when a content item is selected. The standard back button appears to let users navigate backward through the columns.

Controlling Compact Column (iOS 17+)

Starting with iOS 17, you can use the preferredCompactColumn parameter to specify which column appears when the app launches in a compact layout. This gives you control over the initial user experience on iPhone.

Swift 
// Show the detail column first on compact devices
@State private var preferredColumn =
    NavigationSplitViewColumn.detail

var body: some View {
    NavigationSplitView(
        preferredCompactColumn: $preferredColumn
    ) {
        SidebarView()
    } detail: {
        DetailView()
    }
}

Platform Differences

NavigationSplitView behaves differently across Apple's platforms. Understanding these differences is essential for building polished cross-platform apps.

iOS (iPhone)

iPhone
  • Portrait: Always compact -- behaves like a NavigationStack with push/pop.
  • Landscape (Max models): Gets regular horizontal size class. The sidebar can appear as an overlay alongside the detail.
  • Landscape (standard models): Still compact -- single column stack.
  • The sidebar toggle button is automatically provided in the toolbar.

iPadOS

iPad
  • Full screen (portrait or landscape): Multi-column layout with all columns visible.
  • Landscape: Full three-column layout possible with sufficient width.
  • Portrait: Often shows two columns; sidebar may overlay.
  • Slide Over / 1/3 split: Collapses to compact (stack) layout.
  • 1/2 split: May show two columns depending on iPad model.
  • Users can drag column dividers to resize (when using .balanced style).

macOS

Mac
  • Always multi-column -- never collapses to stack.
  • Sidebar has a translucent material background by default, matching macOS design language.
  • The sidebar can be toggled via the toolbar button or Cmd+Ctrl+S.
  • Column widths are resizable by dragging dividers.
  • The sidebar uses .listStyle(.sidebar) automatically, giving grouped section headers.
  • Window resizing dynamically adjusts column proportions.
macOS Sidebar Material

On macOS, the sidebar column automatically gets a translucent material background that shows the desktop behind the window. This is the expected macOS design pattern. If you need to customize it, you can apply .background() to the sidebar content, but be aware that this may look out of place compared to native Mac apps.

Tradeoff Analysis

✓ Advantages

  • Automatic adaptation -- one codebase works across iPhone, iPad, and Mac
  • Declarative selection binding -- clean state-driven column coordination
  • Platform-native behavior -- sidebar overlays, translucent materials, toggle animations all come free
  • Composable with NavigationStack -- embed stacks in columns for deeper navigation
  • Column visibility control -- programmatic show/hide of columns
  • Replaces UISplitViewController -- much less boilerplate than UIKit
  • Future-proof -- Apple's recommended approach for multi-column navigation

✗ Limitations

  • iOS 16+ only -- cannot use on older deployment targets
  • Column width control is unreliable -- especially for the sidebar on iPadOS
  • Limited to 2-3 columns -- no support for 4+ column layouts
  • Compact collapse can be surprising -- navigation state needs careful management when switching between size classes
  • Selection binding quirks -- needs explicit .onChange to cascade resets
  • State restoration complexity -- deep link restoration requires restoring all selection bindings in order
  • No custom transitions -- column show/hide animations are system-controlled

When to Use NavigationSplitView

Use NavigationSplitView when:

  • Your app has a sidebar-driven navigation pattern (categories, mailboxes, folders)
  • You are targeting iPad and/or Mac where multi-column layouts shine
  • You want automatic adaptation to iPhone without maintaining separate navigation code
  • Your content follows a master-detail relationship (list of items + item detail)
  • You are building a document-based app, mail client, file browser, or settings interface
  • You want to support macOS Catalyst or native macOS with proper sidebar behavior

Use NavigationStack instead when:

  • Your app is iPhone-only with no plans for iPad/Mac
  • Navigation is linear and sequential (onboarding, checkout, wizard)
  • You need deep push/pop hierarchies without column structure
  • The app has a tab-based structure where each tab has its own stack

Use both together when:

  • You need multi-column layout plus deep drill-down within the detail column
  • You are building a universal app that should feel native on every form factor
  • Your detail views have their own sub-navigation that goes beyond the column model

Best Practices

1. Always provide a placeholder for empty selections

Use ContentUnavailableView (iOS 17+) or a custom placeholder in the detail column when nothing is selected. On iPad, the detail column is visible even before the user makes a selection.

2. Reset child selections when parent changes

In three-column layouts, attach .onChange(of: parentSelection) and set the child selection to nil. This prevents showing stale detail from a different category.

3. Make selection types Hashable

The List(selection:) API requires Hashable conformance. Use Identifiable and Hashable on your model types for best results with value-based NavigationLink.

4. Use .balanced style for productivity apps

If your sidebar and detail are equally important (like Mail or a file browser), use .navigationSplitViewStyle(.balanced) so both columns share space. Use the default .prominentDetail for media-focused apps where the detail is primary.

5. Test on multiple form factors

Always test your NavigationSplitView on iPhone (portrait and landscape), iPad (full screen, split view, and Slide Over), and Mac if applicable. The automatic adaptation can produce unexpected layouts if your state management is not robust.

6. Be careful with NavigationStack inside columns

NavigationSplitView automatically wraps column content in a NavigationStack. Only add an explicit NavigationStack if you need to manage a NavigationPath for programmatic push/pop within that column.

7. Support keyboard shortcuts on Mac

On macOS, users expect keyboard navigation. The sidebar toggle (Cmd+Ctrl+S) works automatically. Consider adding .keyboardShortcut() to common actions in your detail view.

Feature Minimum Version Notes

ContentUnavailableView iOS 17, macOS 14 System-provided empty state placeholder
NavigationLink(value:) iOS 16, macOS 13 Value-based navigation links (used with selection binding)
.onChange(of:) (new form) iOS 17, macOS 14 Simplified closure without old/new values. Pre-iOS 17 uses .onChange(of:perform:)
Migration from NavigationView

NavigationView was deprecated in iOS 16. If you are migrating, replace NavigationView { ... }.navigationViewStyle(.columns) with NavigationSplitView { ... }. For stack-style NavigationView, use NavigationStack instead. Apple provides a migration guide for detailed instructions.
