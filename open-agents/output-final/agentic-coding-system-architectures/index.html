How It Works

  • Workers cannot share insights discovered mid-task
  • Task decomposition quality depends entirely on orchestrator
  • Merge conflicts when workers modify shared files

Swift/iOS Application

  • Challenge: SwiftUI blurs view/logic boundaries (@State, @Binding in views)
  • Use git worktrees so each worker operates on its own checkout
  • Best for large features spanning many independent files

  • UI Agent for SwiftUI views, Logic Agent for ViewModels, Test Agent for XCTest suites
  • Each agent has deeply focused instructions with domain-specific patterns
  • Adding new conventions to one domain doesn't bloat other agents' contexts

Benefits

  • Each agent can have deeply focused instructions with domain-specific examples

Swift/iOS Application

  • Works better with strict architectures (TCA, Clean Architecture) than vanilla SwiftUI
  • TCA's Reducer/View split gives clearer agent boundaries, but TCA itself is harder for AI
  • Consider feature-centric decomposition instead: one agent owns view + viewmodel + tests for one feature

Decision Guide: When to Use Multi-Agent

Anthropic identifies three specific scenarios where multi-agent approaches pay off. If none apply, stick with a single agent.

Signal 1: Context Pollution

When an agent accumulates irrelevant information from one subtask that degrades performance on subsequent tasks. Separate agents with isolated contexts perform better.

Example: Agent researching networking code then trying to build UI — networking context pollutes UI decisions.

Signal 2: Parallelization Need

Multiple independent directions need to be pursued simultaneously. The benefit isn't speed but thoroughness — parallel agents cover more ground than a single agent working within context limits.

Example: Implementing 5 independent SwiftUI views for different screens simultaneously.

Signal 3: Tool Overload

When agents manage 20+ tools, selection accuracy suffers. Breaking work across specialized agents with focused toolsets resolves this.

Example: Agent needs Xcode build tools, network mocking tools, UI testing tools, accessibility tools, and linting tools all at once.

Key Insight: Context-Centric vs. Problem-Centric Decomposition
Anthropic strongly recommends dividing work by context boundaries, not problem type.

Problematic (Problem-Centric): UI Agent, Logic Agent, Test Agent — creates constant coordination overhead because each needs to understand the others' work.

Better (Context-Centric): Feature A Agent (owns view + viewmodel + tests for Feature A), Feature B Agent — groups work requiring shared understanding together.

Context Management Strategies

How you manage what each agent sees is arguably more important than which architecture you choose. "Curating what the model sees so that you get a better result" is the defining skill.

Progressive Disclosure

Keep CLAUDE.md lean with only universal instructions. Reference detailed files that agents load on-demand. Agent-specific files in .claude/agents/ with YAML frontmatter.

CLAUDE.md (lean) → refs → .claude/agents/swiftui.md
                          → .claude/agents/testing.md
                          → docs/architecture.md

File-Based Handoffs

Each agent saves output to distinct files. Subsequent agents read only the artifacts they need. Creates an audit trail for debugging.

Agent A → writes 01-spec.md
Agent B → reads 01-spec.md, writes 02-impl.md
Agent C → reads 02-impl.md, writes 03-review.md

Model Routing

Use powerful models for planning, efficient models for execution. Set via CLAUDE_CODE_SUBAGENT_MODEL environment variable.

Opus → Planning, architecture decisions
Sonnet → Implementation, code generation
Haiku → Verification, linting, simple checks

Recommended file organization for Swift iOS/Mac projects using agents.

Anti-Pattern: Context Explosion
When parent agents pass their full history to sub-agents, and sub-agents do the same, context grows exponentially. Pass only artifacts and clear contracts — never full conversation histories.

Key Findings

  • MVVM: Good balance. Clear separation without excessive complexity.
  • TCA: ~80 min, 3 errors. Multiple syntax versions confuse AI. But strict separation helps agent decomposition.
  • Clean Architecture: Clear boundaries help domain splitting but more files for AI to navigate.
"Your choice of architecture should primarily be driven by what you believe is the right architecture for maintaining the software you are building, and not by the AI tool you're using."

SwiftUI-Specific Multi-Agent Challenges

Blurry Layer Boundaries

SwiftUI tightly couples view code with state management. @State, @Binding, @Observable live in view files, making it hard to cleanly split "UI Agent" from "Logic Agent". UIKit had clearer MVC boundaries.

Xcode 26.3 Integration

Apple now supports agentic coding with Claude and Codex directly in Xcode. Agents can search docs, update project settings, and capture Previews to visually verify their work. This changes the tooling landscape significantly.

TCA Complexity

TCA's strict Reducer/View split theoretically enables cleaner agent decomposition, but AI struggles with TCA's complex patterns and multiple published syntax versions (old WithViewStore vs new @Bindable).

Verification Advantage

Swift's strong type system + compiler provides excellent automated verification. swift build, swift test, SwiftLint, and Xcode Previews create tight feedback loops ideal for verification subagents.

Recommended Agent Setup for Your Swift Team

💡
Pragmatic Recommendation for a Small Swift Team
Based on the research, consider a hybrid approach rather than pure domain-specialized agents:

1. Feature-centric implementation agents (not layer-centric): Each agent owns a complete feature (view + viewmodel + tests). This avoids coordination overhead while still enabling parallelization.

2. Specialized review agents (this is where domain-specialization shines): Architecture reviewer, memory/concurrency reviewer, accessibility reviewer. These run in parallel with fresh context after implementation.

3. Verification subagent: Build + test + lint runner as a tight feedback loop during implementation.

This gives you the benefits of your domain-specialized idea (focused agent instructions, customizable per concern) while avoiding the key drawback (coordination overhead between UI/Logic/Test agents that need shared context).

Common Pitfalls & Anti-Patterns

1. Over-Fragmentation
Splitting agents too finely creates coordination overhead exceeding specialization benefits. Layer-based splitting (UI/Logic/Data) when features cross all layers creates constant handoff problems.
Fix: Divide by context boundaries, not problem type.
2. Context Explosion
Passing full conversation history from parent to child agents triggers exponential growth. Context windows flooded with irrelevant data cause the model to fixate on past patterns.
Fix: Pass only artifacts and clear contracts, never full histories.
3. The Telephone Game
Information degrades at each agent handoff. By the 4th agent in a chain, critical nuances from the original requirement are lost completely.
Fix: Keep chains short. Include original requirements at each stage.
4. Conflicting Instructions
Multiple agents with overlapping responsibilities make incompatible decisions. Two "code quality" agents with different standards create contradictory feedback loops.
Fix: Clear scope boundaries and single-responsibility per agent.
5. Premature Multi-Agent
Teams spend months building elaborate systems when improved prompting on a single agent would achieve equivalent results. The most common and costly mistake.
Fix: Start simple. Add agents only when specific evidence supports it.
6. No Human Gates
Fully autonomous agent swarms that write, review, and merge without human checkpoints. Quality degrades as agents optimize for passing each other's checks, not actual correctness.
Fix: Always include human gates at critical decision points.
7. Stale Memory
Persistent memory accumulates outdated patterns, deprecated APIs, or superseded decisions. Agents follow stale instructions confidently and generate outdated code.
Fix: Regular memory hygiene. Version-stamp persistent instructions.
8. Tests as Afterthought
Agents write code first and tests second (or skip tests). Without the tight feedback loop of test → implement → verify, quality drops dramatically.
Fix: Test-first or test-alongside workflows. Strict coverage gates on PRs.

Cautionary Evidence

  • SWE-bench: 74.4% on bug fixes but only 11% on complex features
  • Code Smells and Security Hotspots increase with agent PR volume
🔑
Quality Depends On Infrastructure, Not Architecture
Quality outcomes depend heavily on: (1) pre-existing test infrastructure and automated feedback loops, (2) code health of the existing codebase (agents perform best in healthy code), (3) human oversight and review quality, and (4) specificity of agent instructions. AI performs best in healthy code — Code Health score of 9.5-10.0 is recommended before agent deployment.
